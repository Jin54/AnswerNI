## 문의 요약
결재(승인) 저장이 간헐적으로 실패하고 느려지며, lock 관련 로그가 관찰된다는 문의입니다.

## 원인 분석 (로그 근거 인용)
`demo/logs/db.log` 에서 락 경합 정황이 명확히 확인됩니다.

- `2026-07-08 16:40:31 WARN [db] slow query 30021ms UPDATE policy SET ... (T1 holds row lock)` — 약 30초짜리 장기 쿼리(T1)가 행 잠금을 점유.
- `2026-07-08 16:40:35 ERROR [db] tx T2 lock wait timeout exceeded; try restarting transaction` — 후속 트랜잭션(T2)이 락 대기 후 타임아웃.
- `2026-07-08 16:40:35 ERROR [adm] approve save failed doc=3120 approver=[EMAIL_1] reason=lock_wait_timeout` — 락 타임아웃으로 결재 저장 실패(문서 doc=3120, 승인자 [EMAIL_1]).
- `2026-07-08 16:41:10 WARN [db] 6 transactions waiting on row lock (table=approve)` — approve 테이블에서 다수(6건) 트랜잭션이 락 대기.
- `2026-07-08 16:45:01 INFO [db] lock contention cleared` — 이후 락 경합 해소.

즉, 장시간 실행되는 트랜잭션(T1)이 행 잠금을 오래 점유하는 동안 결재 저장 등 후속 쿼리가 락을 기다리다 타임아웃되어 저장이 실패·지연된 것이 원인입니다. (참고: `admin-approve.log` 및 `dlp-approve-api.log` 에서는 'lock'/'ERROR' 해당 줄이 없어, 근거는 db.log 에 집중되어 있습니다.)

## 유사 사례 (Jira)
- **SUP-148 — DB 락 대기/타임아웃으로 결재·정책 저장이 지연됨** (고객사 B, 2026-02-25): 본 건과 거의 동일한 사례. db 로그에 `slow query 30021ms ... holds row lock`, `lock wait timeout exceeded`, `transactions waiting on row lock` 이 기록되었고, 장기 트랜잭션의 행 잠금 장기 점유가 원인으로 규명됨. 조치 후 `lock contention cleared` 확인.

## 해결 방법
1. 장기 실행 쿼리(예: `UPDATE policy ...`) 튜닝 — 실행계획 점검, 인덱스 보강으로 실행 시간을 단축.
2. 락/커넥션 풀 타임아웃 값 조정으로 대기 폭주 완화.
3. 장기 트랜잭션을 분할 처리하여 행 잠금 점유 시간을 단축.
4. 슬로우 쿼리 모니터링을 추가해 락 경합을 조기 탐지.

## 권장 후속 조치
- 위 조치 적용 후 `lock wait timeout` 재발 여부와 `lock contention cleared` 로그로 정상화 확인.
- approve 테이블 대상 락 대기(waiting on row lock) 건수를 대시보드로 상시 모니터링.
- 정책/결재 대량 저장 작업은 피크 시간대를 피해 배치 분산 처리 권장.
