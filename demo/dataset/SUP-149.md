## 문의 요약
특정 PC들이 시간이 지날수록 점점 느려진다는 문의. 에이전트(제품) 자체의 문제인지 분석 요청.

## 원인 분석 (로그 근거 인용)
에이전트 성능 로그(`demo/logs/agent-perf.log`)에서 시간 경과에 따른 명확한 메모리 누수 정황이 확인됩니다.

- `2026-07-08 15:00:00 WARN [perf] rss=1180MB cpu=18% (growing)`
- `2026-07-08 18:00:00 WARN [perf] rss=1890MB cpu=27% handles=8421`
- `2026-07-08 20:30:00 ERROR [perf] high memory: rss=2360MB, scan module holding buffers`

시간이 흐를수록 프로세스 RSS(메모리 사용량)가 1180MB → 1890MB → 2360MB로 지속 증가하고, 핸들 수도 8421개로 늘어나는 등 자원이 해제되지 않고 누적되는 전형적인 메모리 누수 패턴입니다. 특히 마지막 ERROR 로그의 `scan module holding buffers` 문구는 **검사(scan) 모듈이 버퍼를 붙잡고 해제하지 않는 것**이 누수의 근원임을 가리킵니다. 메모리 압박이 누적되면 시스템 전반이 느려지므로, 문의하신 'PC가 시간 지날수록 느려짐' 증상과 일치합니다.

> 참고: 시스템 로그(`demo/logs/system.log`)에는 디스크 부족(`No space left on device`)으로 인한 감사 로그 기록 실패·503 오류가 별도로 존재하나, 이는 이번 '점진적 성능 저하' 문의와는 직접적 관련이 없는 별개 이슈입니다.

## 유사 사례 (Jira)
- **SUP-149 — 에이전트 메모리 누수로 PC 성능 저하** (고객사 F, 2026-06-18): 이번 문의와 거의 동일한 사례입니다. agent-perf 로그에서 RSS가 180MB→2360MB로 지속 증가하고 `high memory: ... scan module holding buffers`가 기록됨. 특정 검사 모듈에서 핸들/버퍼가 해제되지 않아 메모리 누수가 누적된 것이 원인으로 규명되었습니다.

## 해결 방법
SUP-149의 조치 내역을 그대로 적용할 것을 권장합니다.

1. **누수 모듈 패치 적용**: 검사(scan) 모듈의 버퍼/핸들 미해제를 수정한 패치를 적용합니다(근본 원인 제거).
2. **임시 완화**: 패치 적용 전까지 에이전트 서비스 재기동 스케줄을 운영해 메모리를 주기적으로 회복시킵니다. (SUP-149에서 재기동 후 RSS 190MB로 정상 회복 확인)
3. **모니터링 추가**: 메모리 사용량 임계치 알림을 설정해 누수를 조기에 감지합니다.

## 권장 후속 조치
- 대상 PC들의 에이전트 버전을 확인하고, 누수 패치가 반영된 최신 버전으로 일괄 업데이트.
- 패치 적용 전 임시 대책으로 야간 등 유휴 시간대 에이전트 재기동 스케줄 배포.
- RSS/핸들 수 임계치 모니터링 및 알림 구성으로 재발 조기 감지.
- (별개) system.log의 디스크 부족 오류는 감사 로그 유실·503을 유발하므로, 해당 서버의 디스크 용량 확보를 별도 티켓으로 처리 권장.
