## 문의 요약
특정 시간대에 관리서버 응답이 느려진다는 문의. 로그를 통해 성능 저하가 발생한 시점과 원인 정황을 조사했다.

## 원인 분석 (로그 근거 인용)
로그 조사 결과, 특정 시간대 관리서버 지연을 유발할 수 있는 정황이 복합적으로 확인되었다.

1. **DB 행 잠금(row lock) 경합 — 가장 유력한 직접 원인**
   - `demo/logs/db.log`
     - `2026-07-08 16:40:31 WARN [db] slow query 30021ms UPDATE policy SET ... (T1 holds row lock)`
     - `2026-07-08 16:41:10 WARN [db] 6 transactions waiting on row lock (table=approve)`
   - 장기 실행 트랜잭션(T1)이 30초 이상 행 잠금을 점유하고, 그 사이 6개 트랜잭션이 락 대기 상태에 빠졌다. 특정 시간(16:40 전후)에 후속 쿼리가 일제히 대기하면서 관리서버 응답이 느려지는 전형적 패턴이다.

2. **디스크 100% / 로그 적재 실패 — 새벽 시간대 지연·503 원인**
   - `demo/logs/system.log`
     - `2026-07-08 03:41:10 ERROR [sys] write failed: No space left on device (/var/log/app.log)`
     - `2026-07-08 03:41:10 ERROR [app] failed to append audit log: No space left on device`
     - `2026-07-08 03:45:00 ERROR [api] 503 Service Unavailable (audit write failed)`
   - 새벽 03:41 전후 디스크 공간 부족으로 감사 로그 적재가 실패하고 일부 API가 503을 반환했다. '특정 시간(새벽)'에 느려진다면 이 정황이 직접 원인일 수 있다.

3. **관리 프로세스 메모리 증가 추세 — 잠재적 악화 요인**
   - `demo/logs/agent-perf.log`
     - `2026-07-08 15:00:00 WARN [perf] rss=1180MB cpu=18% (growing)`
     - `2026-07-08 18:00:00 WARN [perf] rss=1890MB cpu=27% handles=8421`
   - 오후 시간대에 걸쳐 RSS가 1180MB→1890MB로 증가(growing)하고 핸들 수가 8421까지 늘었다. 메모리 누수/핸들 누수 가능성이 있어 시간이 지날수록 응답 저하를 가중시킬 수 있다.

## 유사 사례 (Jira)
- **SUP-148 (고객사 B, 2026-02-25) — DB 락 대기/타임아웃으로 결재·정책 저장 지연**: `slow query 30021ms ... holds row lock`, `transactions waiting on row lock` 로그가 본 건과 동일. 장기 실행 트랜잭션이 행 잠금을 오래 점유해 후속 쿼리가 락 대기·타임아웃된 것이 원인이었다. → 위 원인 1과 정확히 일치.
- **SUP-145 (고객사 F, 2025-08-19) — 디스크 100%로 로그 적재 실패 및 서비스 지연**: `No space left on device`, `503 Service Unavailable` 로그가 본 건 새벽 시간대 정황과 일치. 로그 로테이션 미설정으로 파티션이 가득 찬 것이 원인이었다. → 위 원인 2와 일치.
- 메모리 증가 추세(원인 3)에 대한 유사 Jira 사례는 검색되지 않았다.

## 해결 방법
1. **DB 락 경합 (SUP-148 준용)**
   - 30초 이상 실행되는 `UPDATE policy` 등 장기 쿼리를 튜닝하고, 장기 트랜잭션을 분할 처리하여 잠금 점유 시간을 단축한다.
   - 락/커넥션 풀 타임아웃을 조정하고 슬로우 쿼리 모니터링을 추가한다. 조치 후 `lock contention cleared` 확인.
2. **디스크 부족 (SUP-145 준용)**
   - 오래된 로그를 정리해 공간을 확보하고, `logrotate`를 일 단위·보관 14일로 설정한다.
   - 디스크 사용률 85% 초과 시 알림 임계치를 등록해 재발을 방지한다.
3. **메모리 증가 추세**
   - 관리 프로세스의 RSS·핸들 수를 지속 모니터링하고, 누수 재현 시 힙/핸들 프로파일링으로 원인 모듈을 특정한다. 임시 완화책으로 예약 재기동을 고려한다.

## 권장 후속 조치
- 고객에게 '느려지는 특정 시간대'를 정확히 확인한다(예: 새벽 03시대인지, 업무시간 16시대인지)에 따라 원인 1·2의 우선순위가 달라진다.
- 원인 1(락)·2(디스크)는 SUP-148·SUP-145 조치 이력이 있으므로 우선 적용한다.
- 슬로우 쿼리·디스크 사용률·프로세스 메모리에 대한 상시 임계치 알림을 등록하여 재발을 조기 감지한다.
