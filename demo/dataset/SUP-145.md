## 문의 요약
새벽부터 관리서버 응답이 느려지고, 로그가 더 이상 적재되지 않는다는 문의.

## 원인 분석 (로그 근거 인용)
로그 조사 결과, 근본 원인은 **로그 파티션(/var/log) 디스크 full** 로 확인된다.

- `system.log` (WARN): 새벽 이전부터 사용률이 상승하다가 완전히 소진됨
  - `2026-07-08 02:14:52 WARN [sys] disk usage /var/log at 92%`
  - `2026-07-08 03:42:33 WARN [sys] disk usage /var/log at 100%`
- `system.log` (ERROR): 디스크가 가득 차 로그/감사 기록 쓰기가 실패하고, 이로 인해 API 가 503 을 반환
  - `2026-07-08 03:41:10 ERROR [sys] write failed: No space left on device (/var/log/app.log)`
  - `2026-07-08 03:41:10 ERROR [app] failed to append audit log: No space left on device`
  - `2026-07-08 03:45:00 ERROR [api] 503 Service Unavailable (audit write failed)`

즉 '로그가 안 쌓임'은 `No space left on device` 로 인한 쓰기 실패이며, '느림/일부 장애'는 감사 로그 쓰기 실패에 연동된 503 응답이 직접적 원인이다. 문의(새벽·느림·로그 미적재) 시점과 로그 타임라인(02~03시대 사용률 92%→100%)이 정확히 일치한다.

### 참고: 문의와 직접 관련은 낮은 부수 관찰
조사 중 아래 항목도 확인되었으나, 이번 새벽 증상의 직접 원인으로 보기는 어렵다(시간대·성격 상이). 별도 모니터링 대상으로만 기록한다.
- `db.log`: `16:40:35` 대 트랜잭션 잠금 대기 타임아웃(`lock wait timeout exceeded`) 및 결재 저장 실패 — 새벽 증상과 시간대가 다름.
- `agent-perf.log`: RSS 가 15시 1180MB → 18시 1890MB 로 증가(handles=8421)하는 메모리 증가 경향 — 지속 관찰 필요.

  - 증상이 현재 문의와 거의 동일: 새벽부터 응답 지연·로그 미적재, `disk usage /var/log at 100%`, `No space left on device`, 일부 API `503 Service Unavailable`.
  - 원인: **로그 로테이션(logrotate) 미설정**으로 대용량 로그가 누적되어 파티션이 가득 참.

## 해결 방법
1. **즉시 조치 — 공간 확보**: `/var/log` 의 오래된/대용량 로그를 정리하여 여유 공간을 확보한다(SUP-145 에서는 약 18GB 확보로 정상화).
2. **서비스 정상화 확인**: 공간 확보 후 로그 append 성공 여부와 API 503 해소를 확인한다.
3. **재발 방지 — 로그 로테이션 설정**: logrotate 를 일 단위·보관 14일 기준으로 구성하여 로그가 무한 누적되지 않도록 한다.

## 권장 후속 조치
- **디스크 사용률 알림 임계치 등록**: `/var/log` 사용률 85% 초과 시 사전 경보(현재는 92%→100% 도달 후에야 장애 인지).
- **DB 잠금 타임아웃 모니터링**: `db.log` 의 16:40 대 `lock wait timeout` / 결재 저장 실패가 반복되는지 별도 추적(장기 실행 트랜잭션·인덱스 점검).
- **에이전트 메모리 증가 추적**: `agent-perf.log` 의 RSS/handle 증가 경향을 관찰해 메모리 누수 여부를 판단.
- 조치 완료 후 사용률 정상화 수치를 기록하고, 본 건을 SUP-145 관련 이슈로 연결해 관리한다.
