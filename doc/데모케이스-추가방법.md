# SUP 케이스를 AnswerNI Demo.dc 에 적용하는 방법

`demo/dataset/SUP-XXX.md`(실제 파이프라인 트레이스)를 스탠드얼론 데모
`AnswerNI-site/AnswerNI Demo.dc.html` 에 케이스로 얹는 절차. SUP-150 을 예시로 정리한다.

> 핵심: Demo.dc 의 `logs` 는 raw 트레이스를 **그대로** 넣는 곳이 아니라, 발표용으로
> **대화체로 재서술**한 버전이다. 단 `mask` 항목의 `raw`/`masked` 는 실제 마스킹 값을
> 그대로 써서 "원본 → 치환"이 화면에 보이게 한다.
>
> Demo.dc 의 갤러리 카드는 **하드코딩된 6장**(card 0~5)이고, 각 카드의
> `onClick="{{ o0 }}"` … `{{ o5 }}` 가 **인덱스로** `cases[0]` … `cases[5]` 상세를 연다
> (`open(i){ this.setState({open:this.cases[i]}) }`). 그래서 케이스를 바꾸려면
> **카드 N 의 정적 HTML** 과 **cases[N] 데이터** 를 짝으로 함께 손대야 한다. 카드가 6장
> 고정이므로, 새 케이스는 보통 **기존 카드 하나를 교체**하는 방식이 자연스럽다
> (예: SUP-150 은 card 0 + cases[0] 의 SUP-140 을 교체함).

---

## 1. Demo.dc 구조

`AnswerNI Demo.dc.html` 안의 `<script type="text/x-dc" data-dc-script>` 블록에
`class Component extends DCLogic { ... cases = [ ... ]; ... }` 가 있다.
`cases` 배열의 각 원소가 데모 카드 1건이며, 스키마는 다음과 같다.

| 필드 | 타입 | 설명 | SUP-XXX.md 매핑 |
|---|---|---|---|
| `sup` | string | 케이스 키 | 파일명/제목의 `SUP-XXX` |
| `cat` | string | 카테고리 배지 | 헤더 `- 카테고리` |
| `title` | string | 카드 제목(짧게) | 헤더 제목 축약 |
| `cust` | string | 고객사 | 헤더 `- 고객사` |
| `date` | string | 접수일(YYYY-MM-DD) | 헤더 `- 접수일` |
| `status` | `{l,d}` | 상태 라벨/점 색 | `{"l":"해결 완료","d":"#22C55E"}` |
| `desc` | string | 한 줄 요약 | 문의/원인 요약 |
| `matchCount` | number | Jira 매치 건수 | 트레이스의 매치 건수 |
| `matchKeys` | string | 매치 키(표시용) | 예 `"SUP-150"` |
| `jiraSummary` | string | 매치 이슈 해결 요약 | Jira `resolution` 축약 |
| `inquiry` | string | 문의 원문 | `## 문의` 본문 |
| `logs` | array | 진행 로그(재서술) | `## 진행상황` 트레이스 → 아래 2절 |
| `jira` | array | 매치 이슈 원문 | `## 유사 Jira 사례` / issues.json |
| `report` | string | 최종 보고서(\r\n) | `## 최종보고서` 전문 |

## 2. `logs` 재서술 규칙

트레이스의 이벤트를 아래 두 종류 항목으로 옮긴다.

- **대화/행동 로그**: `{"k":"log","s":"remote"|"local","ts":"HH:MM:SS","m":"..."}`
  - `s:"remote"` = 원격 LLM 판단을 **1인칭 대화체**로 재서술
    (예: "먼저 id-verify 로그의 오류부터 확인하겠습니다. ERROR 키워드로 열어 주세요.").
  - `s:"local"` = 로컬 에이전트 행동을 **요약체**로.
    형식 관례: `로그 조회: <파일> ('<키워드>') → N줄 / M자`,
    `Jira 검색: '<쿼리>' → 매치 N건 (키)`, `Jira 결과 수신: <키>`,
    `에이전트 종료 (stop_reason=end_turn · 누적 도구 호출 N회)`.
- **마스킹 로그**(중요): `{"k":"mask","ts":"HH:MM:SS","raw":"<원본>","masked":"<치환>"}`
  - `raw` 는 로그 원문(PII 평문), `masked` 는 `app/pii.py` 로 치환한 결과
    (`[EMAIL_n]`·`[RRN_n]`·`[PHONE_n]`·`[IP_n]`). 이 두 값의 차이가
    데모의 "원본 → 치환" 연출이 된다. **반드시 실제 마스킹 값**을 넣는다.

마스킹 값은 실제 함수로 뽑는다(작명·번호가 정확해짐):

```bash
PYTHONIOENCODING=utf-8 ./.venv/Scripts/python.exe -c "
import sys; sys.path.insert(0,'.')
from app import pii, tools
pii.reset()
raw = tools.execute_tool('read_file', {'path':'demo/logs/id-verify.log','keyword':'ERROR'})
print(raw); print('----'); print(pii.mask(raw))
"
```

## 3. 적용 위치 — 카드 + cases 를 짝으로

**(A) 정적 카드 교체**: `<!-- card N -->` 블록 안의 배지(`SUP-1XX` 2곳)·제목·요약·
고객사·접수일·Jira 요약을 새 케이스 값으로 바꾼다. `onClick="{{ oN }}"` 는 그대로 둔다
(인덱스 유지).

**(B) cases[N] 교체**: `cases = [ … ];` 에서 같은 인덱스 N 번째 객체를 새 케이스로 바꾼다.

> 새 케이스를 **추가**(교체 아님)하려면 카드도 1장 더 만들어야 하며, 카드 배치를 그리는
> 캐러셀 좌표 로직(`o0..o5` 핸들러·카드 수)까지 손봐야 하므로 권장하지 않는다.
> 6장 중 하나를 교체하는 편이 안전하다.

### ⚠️ re.sub 치명적 gotcha
케이스 객체를 파이썬 `re.sub(pattern, repl, ...)` 로 끼워 넣을 때, **repl 을 문자열로 주면
그 안의 `\n`·`\r`·`\t` 이 실제 제어문자로 해석**되어 JSON/JS 문자열이 깨진다(report 의
`\r\n`, mask 의 `\n` 이 실제 개행이 되어 파싱 불가). 반드시 **함수 replacement** 를 써라:

```python
html, n = pat.subn(lambda m: block + ",\n", html)   # O — 이스케이프 보존
# html = pat.sub(block, html)                         # X — block 의 \n 이 개행으로 터짐
```

## 3-1. ⚠️ 최종 보고서 파일 (`reports/SUP-XXX.html`) — 필수

데모 재생(`playDemo`)은 `buildScript` 가 만든 `reportUrl: 'reports/'+c.sup+'.html'` 을
LiveDemo iframe 에 넘기고, LiveDemo 는 **`sc.reportUrl` 이 있으면 그 HTML 을 iframe 으로
임베드**한다(없을 때만 인라인 `sc.report` 마크다운 렌더 — 그런데 reportUrl 은 항상 세팅됨).
따라서 **`AnswerNI-site/reports/SUP-XXX.html` 파일이 없으면 최종 보고서가 빈 화면으로 뜬다.**

새 케이스를 넣으면 이 리포트 HTML 도 반드시 만든다. 포맷은 기존 `reports/SUP-140.html`
(hero · TOC · stat-grid · logbox · callout · timeline · 부록 A/B) 을 그대로 재사용하고
본문만 케이스 내용으로 교체한다. `logbox` 의 PII 는 **마스킹된 토큰**(`[EMAIL_n]` 등)으로
쓰고, **부록 B(PII 마스킹 처리 내역)** 표에 토큰↔원문 유형을 정리한다.
(SUP-150 은 `SUP-140.html` 의 head/CSS 를 재사용해 생성함 — 스크립트: `gen_report150.py`.)

## 4. 검증

임베드된 JS의 `cases` 배열이 유효 JSON 으로 파싱되는지 확인한다(모든 키가 따옴표라 JSON 호환).

```bash
PYTHONIOENCODING=utf-8 ./.venv/Scripts/python.exe -c "
import json, re
html = open('AnswerNI-site/AnswerNI Demo.dc.html', encoding='utf-8').read()
arr = json.loads(re.search(r'cases = (\[.*?\]);', html, re.S).group(1))
print('cases', len(arr), '건'); [print(' ', c['sup'], sum(1 for l in c['logs'] if l['k']=='mask'),'mask') for c in arr]
"
```

## 5. 재현용 생성 스크립트(선택)

`report` 의 `\r\n`·따옴표 이스케이프를 손으로 하면 실수가 나므로, 케이스 dict 를
`json.dumps(case, ensure_ascii=False, indent=2)` 로 직렬화(=유효 JS 리터럴)한 뒤 4칸
들여쓰기를 덧대(필드 6칸) 해당 인덱스의 케이스와 교체하는 스크립트를 쓰는 것이 안전하다
(3절 gotcha 대로 **함수 replacement** 필수). SUP-150 적용에 쓴 스크립트가 이 방식이며,
`SUP-XXX.md` 의 `## 최종보고서` 이후 본문을 그대로 `report` 로 넣는다.
