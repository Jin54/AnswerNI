# AnswerNI — 로컬 AI 에이전트 기반 기술지원 자동화 사업계획서

- 작성일: 2026-07-09
- 작성자: 김기주
- 상태: 초안 (v0.1)

---

## 1. 개요 (Executive Summary)

- 로컬 AI 에이전트(온디바이스 SLM)와 원격 거대언어모델(LLM)을 결합한 **하이브리드 기술지원 자동화 시스템** 구축
- 사용자는 문의·작업을 1회 요청하고 최종 보고서를 1회 수령. 중간 과정(자료 수집, 분석, 재질의)은 로컬 에이전트와 LLM 간 자동 협업으로 처리
- 핵심 차별점 3가지
  - **비용 절감**: 로컬 SLM 전처리로 LLM 전송 토큰 최소화
  - **보안성**: 개인정보(PII)를 로컬에서 차단·마스킹 후 외부 LLM에 전송 → 원문 유출 원천 차단
  - **업무 효율**: 기술지원 인력의 단순 문의 대응 및 개발자 에스컬레이션 감소

---

## 2. 배경 및 문제 정의

### 2.1 현황 문제점

- 기술지원 문의 처리 시 담당자가 로그, 설정 파일, Jira 이력을 수작업으로 취합 → 건당 처리 시간 과다
- 단순·반복 문의가 개발자에게 에스컬레이션되어 개발 리소스 소모
- 외부 LLM 활용 시 두 가지 장벽 존재
  - **보안**: 로그·문의 내용에 포함된 개인정보(이름, 이메일, 전화번호, 계정정보)를 외부로 전송할 수 없음
  - **비용**: 파일 전문을 그대로 전송하면 토큰 비용 급증 및 컨텍스트 낭비

### 2.2 해결 방향

- 데이터가 외부로 나가기 전 **로컬에서 전처리·선별·마스킹**하는 게이트키퍼 계층 신설
- 판단(두뇌)은 원격 LLM, 실행(손발)과 데이터 통제(검문소)는 로컬 에이전트로 역할 분리

## 3. 서비스 개요

### 3.1 동작 흐름

- 사용자 접점은 웹 UI 2회뿐: **최초 문의 입력 1회 + 최종 보고서 수령 1회**
- 그 사이 로컬 데몬과 원격 LLM이 tool call로 **N회 왕복**하며 자료 수집·분석을 자율 수행 (진행 상황은 UI에 실시간 로그로만 표시)

```
[웹 UI] 사용자: 문의 내용 또는 수행할 작업 입력 (1회)
   │
   ▼
[로컬 데몬]
   1. 로컬 SLM 전처리: 프롬프트 정돈, 관련 파일·Jira 이슈 후보 선별
   2. PII 마스킹 후 LLM에 요청 (사용 가능 도구 목록 동시 선언)
   │
   ▼
┌─────────────── N회 반복 (자율 루프) ───────────────┐
│ [원격 LLM] ── tool_use: "○○ 로그 읽어줘 / Jira △△ 찾아줘" │
│    │                                                │
│    ▼                                                │
│ [로컬 데몬] 도구 실행 → 결과 요약(SLM) → PII 마스킹    │
│             → tool_result 반환, 진행 로그 UI 송출     │
└──────── (LLM이 충분하다 판단 시 루프 종료) ────────┘
   │
   ▼
[원격 LLM] 최종 보고서 생성
   │
   ▼
[웹 UI] 완성 보고서 표시 (1회)
```

#### 예시 시나리오: "고객사 A에서 로그인이 안 된다는 문의"

| 순번 | 주체 | 내용 |
|---|---|---|
| 0 | 사용자 → 웹 UI | "고객사 A에서 어제부터 로그인 실패한다는 문의. 원인 분석 보고서 작성해줘" 입력 |
| 1 | 데몬 (SLM 전처리) | 문의를 정돈된 프롬프트로 변환, 관련 후보 파악: `auth-server.log`, Jira의 로그인 관련 이슈 |
| 2 | 데몬 → LLM | 정돈된 문의 + 도구 목록(파일읽기, Jira검색) 전송 |
| 3 | LLM → 데몬 | tool_use: "`auth-server.log`에서 어제 날짜 오류 부분 읽어줘" |
| 4 | 데몬 → LLM | 로그 읽기 실행 → SLM이 오류 구간만 추출 → PII 마스킹(`user: kim@aaa.com` → `user: [EMAIL_1]`) → tool_result 반환. UI에 "auth-server.log 분석 중..." 표시 |
| 5 | LLM → 데몬 | tool_use: "인증서 만료 오류로 보임. Jira에서 'SSL 인증서' 관련 과거 이슈 검색해줘" |
| 6 | 데몬 → LLM | Jira 검색 실행 → 유사 이슈 `SUP-123` 발견 → 마스킹 후 반환. UI에 "Jira 유사 사례 검색 중..." 표시 |
| 7 | LLM → 데몬 | tool_use: "`SUP-123`의 해결 방법 상세 읽어줘" |
| 8 | 데몬 → LLM | 이슈 본문 반환 (왕복 횟수는 문의 복잡도에 따라 가변, 상한 도달 시 강제 마무리) |
| 9 | LLM → 데몬 | 최종 보고서 생성: 원인(인증서 만료) + 근거(로그 발췌) + 과거 유사 사례 + 조치 방안 |
| 10 | 데몬 → 웹 UI | 보고서 렌더링. 마스킹된 항목은 로컬에서만 원문 복원 가능(옵션) |

- 위 예시는 왕복 3회 사례이며, 단순 문의는 1~2회, 복잡한 분석은 상한(예: 10회)까지 자동 수행
- 사용자는 0번과 10번만 관여 — 중간 과정은 진행 로그로 투명하게 노출되지만 개입 불필요

### 3.2 역할 분담

| 구성요소 | 역할 | 비고 |
|---|---|---|
| 원격 LLM | 판단·추론·보고서 생성, tool call로 추가 자료 요청 | Claude API 등 |
| 로컬 에이전트(데몬) | 대화 루프 소유, 도구 실행, PII 게이트키퍼, 감사 로그 | 단일 코드베이스, Mac/Windows |
| 로컬 SLM (Ollama + Gemma 3n E4B) | ① 요청 전처리 ② 대용량 도구 결과 요약·추출 ③ PII 검출 2차 검증 | 온디바이스, 무과금 |
| 웹 UI | 문의 입력, 진행 상태 표시, 최종 보고서 열람 | 데몬이 localhost 서빙 |

### 3.3 핵심 기능

- **자율 도구 실행 루프**: 표준 tool-use(function calling) 기반. 파일 읽기, Jira 검색 등 읽기 전용 도구부터 시작
- **PII 이중 필터**: 1차 정규식·사전 기반 결정적(deterministic) 필터 + 2차 SLM 검증. LLM으로 나가는 모든 출구(최초 요청 + 모든 tool_result)에 공통 미들웨어로 적용
- **도구 허용 범위(allowlist)**: 허용된 디렉터리·Jira 프로젝트 내에서만 실행. 권한은 로컬이 보유
- **루프 제어**: 최대 반복 횟수·토큰 예산 초과 시 강제 마무리, 실패 반복 시 부분 보고
- **감사 로그**: 읽은 파일, 마스킹 내역, LLM 송신 내용 전량 기록 → 보안 증적 및 디버깅

### 3.4 범용 LLM 도구(Claude Code 등) 대비 차별점

- 어필 우선순위: **① 보안(enabler) → ② 비개발자 특화 → ③ 통제·증적 → ④ 컨텍스트 효율**

| # | 차별점 | 내용 |
|---|---|---|
| ① | 보안 — 도입을 가능하게 하는 enabler | 범용 도구는 읽은 파일 원문을 그대로 외부 API로 전송 → 고객 로그·문의에 포함된 개인정보 때문에 이 업무에 **아예 사용 불가**. AnswerNI는 로컬 마스킹으로 "불가능을 가능"으로 전환 |
| ② | 비개발자 특화 제품 | Claude Code는 개발자용 터미널 도구. AnswerNI의 사용자는 기술지원 인력 — 프롬프트 작성 능력 없이 "문의 입력 → 보고서 수령" 단일 워크플로우로 균일한 품질 확보, 보고서 양식 통일 |
| ③ | 조직 차원 통제·증적 | 도구 allowlist와 감사 로그를 데몬이 **우회 불가능하게 강제**. "개인정보가 안 나갔는가"를 증명 가능 — 개인별 범용 도구로는 불가 |
| ④ | 컨텍스트 효율 | SLM 전처리로 원문 전체가 아닌 관련 구간만 전송 → 토큰 비용 절감 + 노이즈 제거로 분석 정확도 향상 |

- 예상 반론 대응: "Claude Code에 훅(hook)으로 마스킹을 붙이면 되지 않나?" → 개인이 설정하는 범용 도구와 마스킹·allowlist·감사가 제품 차원에서 강제되는 전용 시스템은 보안 보증 수준이 다르며, 애초에 비개발자는 대상이 아님

### 3.5 LLM 백엔드 교체 전략 (사내 GPU 서버 도입 시)

- 데몬·도구·UI·마스킹 계층은 LLM 백엔드에 독립적 — **엔드포인트 교체만으로 Claude API → 사내 GPU 서버 self-host LLM 전환 가능** (벤더 종속 없음)
- 전환 시에도 SLM 전처리(토큰 절감)의 가치는 소멸하지 않고 **절약 대상이 API 요금 → GPU 자원으로 전환**됨
  - 입력 토큰 축소 → prefill 시간 단축 → 첫 토큰 응답 대기시간(TTFT) 감소
  - KV 캐시 메모리 절약 → 동일 GPU로 동시 처리 가능 문의 수 증가 → 하드웨어 증설 지연
  - self-host 모델은 프론티어 대비 소형(컨텍스트 윈도우·긴 문맥 성능 열세) → 관련 구간만 추려주는 전처리가 **정확도 유지에 오히려 더 중요**
- 현실적 운영안: 하이브리드 라우팅 — 단순 문의는 사내 LLM, 고난도 분석은 외부 API. 외부 경로가 남는 한 PII 게이트키퍼 역할도 유지

## 4. 기술 아키텍처

### 4.1 플랫폼 전략

- **로컬 데몬 + 로컬 웹 UI** 구조 채택 (Mac/Windows 단일 코드베이스)
  - 순수 웹앱 배제 사유: 사용자 PC의 파일 접근 및 로컬 Ollama 연동 불가 → 보안 설계 핵심 훼손
  - 플랫폼별 네이티브 앱 배제 사유: 유지보수 비용 2배 대비 이점 없음
- UI는 데몬이 서빙하는 로컬 웹 페이지로 시작, 필요 시 Tauri 래핑으로 데스크톱 앱화 (Electron 대비 경량)
- 진행 상태는 SSE/WebSocket으로 스트리밍 (수십 초~수 분 작업 대응)

### 4.2 기술 스택 (안)

| 계층 | 기술 | 선정 사유 |
|---|---|---|
| 에이전트 코어 | Go 또는 Node.js 또는 Python | 단일 코드베이스 크로스 빌드 |
| 로컬 SLM | Ollama + Gemma 3n E4B (약 4~5GB) | 무과금, 온디바이스, Mac/Windows 지원 |
| 원격 LLM | Claude API (tool use) — 추후 사내 GPU 서버 self-host LLM으로 교체 가능 | 표준 tool-use 루프 내장, 백엔드 독립적 설계 (3.5절) |
| 도구 프로토콜 | MCP(Model Context Protocol) 검토 | 도구를 MCP 서버화 시 기존 클라이언트 재사용 가능 |
| UI | 로컬 웹 (→ 필요 시 Tauri) | 배포 유연성 |
| 배포 | 인스톨러 스크립트 (데몬 + Ollama + 모델 번들) | 첫 실행 시 모델 다운로드 자동화 |

## 6. 추진 계획 (해커톤 → 선정 후 확장)

### 6.1 해커톤 당일 MVP 범위

- 목표: **1일 내 "문의 1회 입력 → 자동 루프 → 최종 보고서" 전체 흐름이 동작하는 데모 + 웹 UI 목업** 완성
- 심사 어필 포인트에 집중: ① 자율 루프가 눈에 보이게 (LLM이 파일을 추가 요청하는 과정 실시간 표시) ② PII 마스킹이 눈에 보이게 (마스킹 전/후 비교)

| 구분 | 항목 | 비고 |
|---|---|---|
| 포함 | tool-use 루프 (파일 읽기 + Jira 검색 도구 2종) | Jira는 실연동 어려우면 목데이터로 대체 |
| 포함 | 정규식 기반 PII 마스킹 (이름·이메일·전화번호) | 결정적 1차 필터만 |
| 포함 | 로컬 웹 UI 목업 (문의 입력 → 진행 로그 스트리밍 → 보고서) | 단일 페이지, 데몬이 서빙 |
| 포함 | Ollama SLM 전처리 1개 지점 (프롬프트 정돈 또는 도구 결과 요약) | 시간 부족 시 최우선 제외 후보 |
| 포함 | 데모 시나리오 (가짜 PII 포함 샘플 로그 + 준비된 문의 1건) | 시연 성공률 확보용 |
| 제외 | SLM PII 2차 검증, 감사 로그 뷰어, allowlist 설정 UI | 선정 후 진행 |
| 제외 | Tauri 래핑, 인스톨러, Windows 빌드 | 데모는 개발 장비 1대로 시연 |
| 제외 | 루프 정교화 (토큰 예산, 부분 보고) | 최대 반복 횟수 하드코딩으로 대체 |

### 6.2 당일 타임라인 (약 9시간 기준)

| 시간 | 작업 | 완료 기준 |
|---|---|---|
| 0~1h | 환경 구축: Ollama 모델 pull(사전 완료 권장), API 키, 프로젝트 뼈대 | 스택 결정 완료 |
| 1~4h | 코어: tool-use 루프 + 파일 읽기 도구 + 정규식 PII 마스킹 미들웨어 | CLI에서 보고서 1건 생성 |
| 4~6h | 확장: Jira 검색 도구(목데이터 가능) + SLM 전처리 1개 지점 | 도구 2종 루프 동작 |
| 6~8h | 웹 UI 목업: 입력 → 진행 로그(SSE) → 마스킹 전/후 표시 → 보고서 | 브라우저 데모 동작 |
| 8~9h | 데모 시나리오 리허설 + 발표 자료 정리 | 시연 2회 연속 성공 |

- 리스크 헤지: Ollama 모델 다운로드(4~5GB)는 해커톤 전날 완료. LLM API 키·Jira 접근 권한도 사전 확인
- 우선순위 원칙: 코어 루프 > PII 마스킹 > UI 목업 > SLM 전처리 순. 시간 부족 시 뒤에서부터 제외

### 6.3 선정 후 로드맵

| 단계 | 내용 | 산출물 |
|---|---|---|
| Phase 1. 정식 코어 | PII 이중 필터(SLM 2차 검증), 감사 로그, allowlist, 루프 제어(토큰 예산·부분 보고) | 보안 증적 가능한 알파 |
| Phase 2. 연동·실측 | Jira 실연동, SLM 전처리 확대, 토큰 절감률·처리 시간 베이스라인 실측 | 정량 지표 보고 |
| Phase 3. 배포 | UI 완성, Mac/Windows 인스톨러(데몬+Ollama+모델 번들), Tauri 래핑 검토 | 설치형 베타 |
| Phase 4. 파일럿·확대 | 기술지원팀 파일럿 → 지표 검증 → 전사 확대, 쓰기 도구는 승인 절차 포함 추가 | 정식 운영 |

## 7. 리스크 및 대응 방안

| 리스크 | 영향 | 대응 |
|---|---|---|
| SLM 단독 PII 검출 누락 | 개인정보 유출 | 정규식·사전 기반 1차 결정적 필터 + SLM 2차 검증 이중화, 감사 로그로 전수 증적 |
| LLM 무한 루프·토큰 폭증 | 비용 초과 | 최대 반복 횟수·토큰 예산 상한, 초과 시 수집분 기반 강제 결론 |
| 도구 오남용 (임의 경로 접근) | 보안 사고 | 디렉터리·Jira 프로젝트 allowlist, 읽기 전용 도구로 시작 |
| 저사양 PC의 SLM 성능 저하 | 응답 지연 | 최소 사양 가이드 수립, SLM 미가용 시 정규식 필터 단독 폴백 모드 |
| 최초 설치 부담 (모델 4~5GB) | 도입 저항 | 인스톨러 자동화, 사내망 모델 미러 서버 검토 |
| 무인 루프 실패 시 무응답 | 신뢰 저하 | 실패 반복 시 루프 중단 후 "수행 내역 + 미해결 사유" 부분 보고 |

## 9. 결론

- 본 사업은 "외부 LLM의 지능 + 로컬 에이전트의 보안·실행력" 결합으로, 보안 우려 없이 LLM을 기술지원 업무에 도입하는 현실적 경로임
- 읽기 전용 도구 기반 무인 루프로 시작하여 리스크를 최소화하고, 파일럿 정량 지표(처리 시간, 토큰 절감, PII 0건) 검증 후 단계적 확대
- 전체 흐름이 표준 기술(tool-use, Ollama)의 조합이므로 **해커톤 1일 내 동작하는 MVP + UI 목업 시연이 가능**하며, 선정 시 보안 증적·배포 체계를 갖춘 정식 버전으로 단계적 확장
