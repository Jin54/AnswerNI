1.개요 (Context)

고객사 F에서 새벽 시간대부터 관리서버의 응답이 느려지고 로그가 정상적으로 적재되지 않는다는 문의를 접수했다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 시스템 로그(logs/system.log)와 부수 로그(db.log, agent-perf.log), 그리고 사내 Jira 이력을 조회·대조하여 도출한 원인 분석과 조치 방안을 정리한 것이다.

고객 문의 원문: "새벽부터 관리서버가 느리고 로그가 안 쌓인대요. 원인 분석해줘"
결론 (한 줄 요약): 로그 파티션 /var/log가 디스크 풀(100%) 상태에 도달하여 로그·감사 기록 쓰기가 실패(No space left on device)했고, 감사 로그 쓰기 실패에 연동된 API가 503 Service Unavailable을 반환했다. 근본 원인은 로그 로테이션(logrotate) 미설정으로 대용량 로그가 누적된 것. 오래된 로그 정리(약 18GB 확보)와 logrotate 구성으로 사용률 41% 정상화.

4.로그 포렌식 분석

근거 로그: logs/system.log. 디스크 사용률 상승 → 쓰기 실패 → 감사 로그 실패 → API 503의 인과 사슬이 02~03시대 타임라인에 그대로 드러난다. 문의(새벽·느림·로그 미적재) 시점과 정확히 일치한다.

5.근본 원인 분석 (RCA)

No space left on device는 파일시스템에 쓸 공간이 없을 때 발생하는 전형적 오류다. 로그 파티션 /var/log가 가득 차(logrotate 미설정으로 대용량 로그 누적) 로그·감사 기록 쓰기가 실패했고, 감사 로그 쓰기 실패에 연동된 API가 503을 반환했다. 고객이 체감한 '로그가 안 쌓임' = 쓰기 실패, '느림/일부 장애' = 503이다.

[02:00 nightly job start] | v [로그 누적 — 로테이션(logrotate) 미설정] ❌ 근본 원인 | v [/var/log 92% (02:14 WARN)] → [/var/log 100% (03:42 WARN)] | v [write failed: No space left on device (/var/log/app.log)] | v [audit log append 실패] → ('로그가 안 쌓임'의 실체) | v [API 503 Service Unavailable (audit write failed)] | v [관리서버 느림 / 일부 장애] ('느림'의 실체) | v [(조치) 오래된 로그 18GB 정리] → [/var/log 41% 정상화]

원인 사슬 (Causal Chain)

계층원인
직접 원인/var/log 디스크 풀(100%) → 로그·감사 기록 쓰기 실패(No space left on device).
근본 원인로그 로테이션(logrotate) 미설정으로 대용량 로그가 무제한 누적되어 파티션이 가득 참(02:14 92% → 03:42 100%).
기여 요인디스크 사용률 사전 경보 임계치 부재 — 92%→100% 도달 후에야 인지. 새벽 무인 시간대라 수동 대응이 08시까지 지연.
연쇄 영향감사 로그 append 실패 → 해당 API가 503 반환 → 관리서버 응답 지연.
타임라인 일치: 문의(새벽·느림·로그 미적재) 시점과 로그 타임라인(02~03시대 92%→100%)이 정확히 일치하여, 본 건이 디스크 풀 사건임을 높은 확신으로 확정할 수 있다.

SUP-145 조치 이력(resolution)

오래된 로그를 정리하여 약 18GB를 확보하고, logrotate를 일 단위·보관 14일로 설정, 디스크 사용률 85% 초과 시 알림 임계치를 등록해 재발을 방지했다. 조치 후 사용률 41%로 정상화되었다.

7.해결 방안

7-1. 즉시 조치 — 디스크 공간 확보

/var/log의 오래된·대용량 로그를 정리하여 공간을 확보한다. SUP-145에서는 약 18GB 확보로 사용률이 즉시 정상화되었다.

# 대용량 로그 상위 확인
du -ah /var/log | sort -rh | head -n 20

# 오래된 로그 정리 (예: 14일 초과 회전 로그 제거)
find /var/log -name "*.log.*" -mtime +14 -delete
find /var/log -name "*.gz" -mtime +14 -delete

7-2. 서비스 정상화 확인

공간 확보 후 로그 append 성공과 API 503 해소를 확인한다. SUP-145 기준 조치 후 /var/log 사용률이 41%로 내려가고 정상 적재가 재개되었다.

  • system.log에서 No space left on device 미재발 확인
  • 감사 로그 append 성공(신규 라인 적재) 확인
  • API 응답이 200 정상으로 복귀했는지 확인 (503 해소)

7-3. 재발 방지 — logrotate 구성

로그 로테이션을 일 단위·보관 14일로 구성하여 로그가 무제한 누적되지 않도록 한다. 아래는 예시 설정이다.

# /etc/logrotate.d/app
/var/log/app.log {
    daily
    rotate 14
    compress
    missingok
    notifempty
    copytruncate
}

7-4. 조치 체크리스트

  • 근본 원인 규명 (logrotate 미설정 → /var/log 디스크 풀)
  • 오래된/대용량 로그 정리로 즉시 공간 확보 (약 18GB)
  • 로그 append 성공 · API 503 해소 확인
  • logrotate 일 단위·보관 14일 구성 및 logrotate -d 문법 검증
  • 디스크 사용률 85% 초과 알림 임계치 등록
  • 조치 후 /var/log 사용률 41% 정상화 확인

8.재발 방지 & 모니터링 권고

  • 디스크 사전 경보: /var/log 사용률 85% 초과 시 사전 경보를 등록한다. 현재는 92%→100% 도달 후에야 인지되어 무인 시간대에 장기 방치되었다.
  • 로그 로테이션 상시화: 모든 주요 로그에 logrotate(일 단위·보관 14일·압축)를 적용하여 누적을 원천 차단한다.
  • DB 락 별도 추적: db.log의 lock wait timeout exceeded(오후 시간대)는 본 건과 무관하나, 결재 저장 실패(doc=3120)를 유발하므로 별도 이슈로 추적한다.
  • 에이전트 리소스 관찰: agent-perf.log의 RSS 증가(1180MB→1890MB, handles=8421)는 메모리·핸들 누수 가능성이 있으므로 추이를 관찰한다.
  • 이슈 연계: 본 건을 SUP-145와 linked issue로 기록해 동일 증상 재문의 시 참조한다.

10.부록 A — 에이전트 분석 파이프라인

본 보고서는 로컬 데몬(도구 실행·PII 마스킹·SLM 요약)과 원격 LLM(Claude Code CLI, claude-opus-4-8)이 결합된 자율 tool-use 루프의 산출물이다. 누적 도구 호출 7회, 종료 사유 end_turn.

11.부록 B — PII 마스킹 처리 내역

로컬 에이전트가 원격 LLM 전달 전 로그 내 개인정보를 토큰으로 치환했다. 원문 PII는 원격 LLM에 전달되지 않는다.
