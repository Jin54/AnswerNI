1.개요 (Context)

고객사 I에서 결재 저장이 자꾸 실패하고 응답이 느리다는 문의가 접수되었다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 db.log·webhook.log·admin-approve.log 3종 로그와 사내 Jira 이력을 교차 조사하여 도출한 원인 분석과 조치 방안을 정리한 것이다.

고객 문의 원문: "결재 저장이 자꾸 실패하고 느려요. db 로그에 lock 관련이 보이는데 봐줄래요?"
결론 (한 줄 요약): 장기 실행 트랜잭션(T1, policy_save)이 approve 테이블 행 잠금을 300초간 점유하여, 후속 결재 저장 트랜잭션(T2)이 락 대기 끝에 lock wait timeout exceeded로 실패하는 것이 근본 원인. 잠금 장기 점유로 커넥션 반환이 지연되어 HikariPool 커넥션 풀 고갈(pool exhausted)로도 확산되었다. 장기 쿼리 튜닝 · 트랜잭션 분할 · 락/커넥션 타임아웃 조정 · 슬로우쿼리 모니터링으로 해결한다.

4.로그 포렌식 분석 (다중 로그 교차)

에이전트가 3종 로그를 교차 조사했다. 결재자 이메일·접속 IP 등 PII는 로컬 에이전트가 원격 LLM 전달 전 [EMAIL_n]·[IP_n] 토큰으로 치환했다(부록 B).

4-1. db.log — 주근거: 행 잠금 점유와 타임아웃

2026-07-08 16:40:01 INFO [db] tx begin id=T1 op=policy_save 2026-07-08 16:40:31 WARN [db] slow query 30021ms UPDATE policy SET ... (T1 holds row lock) 2026-07-08 16:40:35 ERROR [db] tx T2 lock wait timeout exceeded; try restarting transaction 2026-07-08 16:40:35 ERROR [adm] approve save failed doc=3120 approver=[EMAIL_1] reason=lock_wait_timeout 2026-07-08 16:41:10 WARN [db] 6 transactions waiting on row lock (table=approve) 2026-07-08 16:45:00 INFO [db] tx T1 committed after 300s 2026-07-08 16:45:01 INFO [db] lock contention cleared
  • T1(policy_save)이 30021ms 슬로우 쿼리로 행 잠금을 장시간 점유.
  • 그 사이 T2(결재 저장)가 락을 얻지 못하고 lock wait timeout exceeded로 실패(doc=3120).
  • 동시에 6개 트랜잭션이 approve 테이블 행 잠금을 대기 → 저장 지연 체감.
  • T1이 300초 후 커밋되자 lock contention cleared로 경합 해소.

4-2. webhook.log — 연계: 커넥션 풀 고갈로 확산

락 경합이 DB 계층에만 머물지 않고 커넥션 풀(HikariPool)로 전파되었는지 확인하기 위해 커넥션 풀 로그를 추가 조사했다.

2026-07-09 09:31:02 ERROR [db] HikariPool-1 - Connection is not available, request timed out after 30000ms 2026-07-09 09:31:02 WARN [db] HikariPool-1 - active=20 idle=0 waiting=14 (pool exhausted)
  • 행 잠금을 문 트랜잭션이 커넥션을 오래 붙들면서 커넥션이 반환되지 않아 active=20 idle=0으로 풀 고갈.
  • 대기 14건이 request timed out after 30000ms로 실패 — 락 경합이 커넥션 풀 계층으로 전파된 2차 증상.

4-3. admin-approve.log — 결재 저장 경로와 PII

결재 저장 경로가 락 대기 구간과 겹쳐 지연됐는지, 그리고 해당 경로에 PII가 남는지 확인했다.

2026-07-09 14:18:02 INFO [adm] approve console opened approver=[EMAIL_2] ip=[IP_1]
주의: 원본 로그에는 결재자 이메일(approver)과 접속 IP가 평문으로 남아 있었다. AnswerNI 로컬 에이전트가 원격 LLM 전달 전 [EMAIL_1]·[EMAIL_2]·[IP_1]로 치환했다(부록 B). 운영 로깅 자체도 마스킹으로 전환해야 한다.

5.근본 원인 분석 (RCA)

모든 신호가 장기 실행 트랜잭션의 행 잠금 장기 점유라는 하나의 뿌리로 수렴한다. 슬로우 쿼리(30021ms)로 T1이 락을 오래 잡고, 그 락을 기다리던 T2가 타임아웃되며, 반환되지 않은 커넥션이 풀을 고갈시킨다.

[T1: policy_save] slow query 30021ms → approve 행 잠금 300초 점유 ❌ | v [T2: 결재 저장] 락 대기 → lock wait timeout exceeded → approve save failed (doc=3120) | v [대기 누적] 6 transactions waiting on row lock (table=approve) | v [커넥션 반환 지연] HikariPool active=20 idle=0 waiting=14 → pool exhausted → request timed out 30000ms

원인 사슬 (Causal Chain)

계층원인
직접 원인후속 결재 저장(T2)이 행 잠금을 얻지 못하고 lock wait timeout으로 실패.
근본 원인장기 실행 트랜잭션(T1, policy_save)이 행 잠금을 300초간 점유 — 슬로우 쿼리(30021ms)와 넓은 트랜잭션 경계.
확산 경로잠금 점유로 커넥션 반환 지연 → HikariPool 커넥션 풀 고갈(active=20 idle=0 waiting=14) → 대기 요청 30s 타임아웃.
기여 요인락/커넥션 타임아웃 설정 부재·과다, 트랜잭션 분할 미적용, 슬로우쿼리 모니터링 부재.
확산 관점: 단일 슬로우 쿼리의 락 점유가 결재 저장 실패 → 락 대기 누적 → 커넥션 풀 고갈 → 서비스 전반 응답 저하로 연쇄 전파된다. 증상(타임아웃)만 완화하지 말고 락 점유 시간 자체를 줄이는 근본 조치가 필요하다.

6.유사 사례 (Jira 매칭)

에이전트가 로그의 핵심 문구 'lock wait timeout'으로 사내 Jira를 조회해 동일 원인의 과거 이슈 SUP-148을 회수했다(자연어 문장 검색 '결재 저장 실패'는 0건 → 키워드 정제 재검색으로 매칭).

SUP-148 조치 이력(resolution)

장기 실행 쿼리를 튜닝하고 락/커넥션 풀 타임아웃을 조정, 슬로우 쿼리 모니터링을 추가. 장기 트랜잭션 분할 처리로 잠금 점유 시간을 단축. 조치 후 lock contention cleared 확인.

7.해결 방안

7-1. 근본 해결 — 락 점유 시간 단축

가장 중요한 조치는 policy_save의 장기 쿼리 튜닝과 트랜잭션 분할이다. 슬로우 쿼리(30021ms)를 인덱스·조건 재검토로 단축하고, 넓은 트랜잭션 경계를 잘게 나눠 행 잠금 점유 시간을 최소화한다. 커넥션 반환도 빨라져 풀 고갈까지 함께 완화된다.

7-2. 병행 — 락/커넥션 타임아웃 조정 & 모니터링

락 대기와 커넥션 풀 타임아웃을 합리적 값으로 설정해 무한 대기·연쇄 지연을 방지하고, 슬로우 쿼리·락 대기·풀 고갈 패턴을 알림 규칙으로 등록한다.

# ❌ 현재 — 하나의 넓은 트랜잭션이 UPDATE를 30초간 붙들며 행 잠금 장기 점유
BEGIN;  UPDATE policy SET ... ;  -- 30021ms, T1 holds row lock
        -- ... 다수 후속 작업 ...
COMMIT; -- 300s 후 커밋 → 그 사이 T2 lock wait timeout

# ✅ 개선 — 쿼리 튜닝 + 트랜잭션 분할 + 타임아웃 가드
SET lock_timeout = '3s';            -- 무한 락 대기 차단
SET idle_in_transaction_session_timeout = '10s';
BEGIN;  UPDATE policy SET ... WHERE id = ?;  -- 인덱스 조건으로 스캔·점유 최소화
COMMIT; -- 짧게 끊어 행 잠금 즉시 반환

7-3. 배포 체크리스트

  • 근본 원인 규명 (장기 트랜잭션 행 잠금 점유 → lock wait timeout → 커넥션 풀 고갈)
  • policy_save 장기 쿼리 튜닝(인덱스/조건) 후 실행 시간 단축 확인
  • 장기 트랜잭션 분할 적용으로 행 잠금 점유 시간 최소화
  • 락 대기 / 커넥션 풀 타임아웃 값 조정
  • 슬로우쿼리·락 대기·pool exhausted 모니터링 알림 등록
  • 배포 후 db.log에서 lock wait timeout / pool exhausted 미재발 확인

8.재발 방지 & 모니터링 권고

  • 모니터링 규칙: slow query NNNNNms · lock wait timeout exceeded · N transactions waiting on row lock · HikariPool ... pool exhausted 패턴을 알림으로 등록해 조기 감지한다.
  • 트랜잭션 경계 점검: 장기 트랜잭션 경계 코드를 리뷰하고, 배치성 UPDATE는 트랜잭션 분할을 적용한다.
  • 타임아웃 표준화: lock_timeout·커넥션 풀 타임아웃을 표준값으로 설정해 무한 대기와 연쇄 지연을 차단한다.
  • PII 로깅 정리: 결재/커넥션 로그의 결재자 이메일·접속 IP 등 PII는 마스킹 로깅으로 전환한다.
  • 이슈 연계: 본 건을 SUP-148과 linked issue로 기록해 동일 증상 재문의 시 참조한다.

10.부록 A — 에이전트 분석 파이프라인

본 보고서는 로컬 데몬(도구 실행·PII 마스킹·SLM 요약)과 원격 LLM(Claude Code CLI, claude-opus-4-8)이 결합된 자율 tool-use 루프의 산출물이다. 누적 도구 호출 6회(로그 조회 4 · Jira 검색 2), 종료 사유 end_turn.

11.부록 B — PII 마스킹 처리 내역

로컬 에이전트가 원격 LLM 전달 전 로그 내 개인정보를 토큰으로 치환했다. 같은 값은 같은 토큰으로 매핑된다. 원문 PII는 원격 LLM에 전달되지 않는다.
