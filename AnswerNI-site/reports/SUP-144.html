1.개요 (Context)

고객사 D(의료)에서 Mac 에이전트가 표시하는 차단/안내 알림 팝업의 노출 시간(화면에 떠 있는 시간)을 관리자가 임의로 조절하고 싶다는 문의를 접수했다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 관련 로그와 사내 Jira 이력을 조회·대조하여, 이 문의가 장애(fault)가 아닌 기능 개선 요청(enhancement)임을 확인하고 현황과 대응 방안을 정리한 것이다.

고객 문의 원문: "고객사에서 알림 팝업 시간을 조절하고 싶다는데, 지금 가능한지랑 대응안 정리해줘"
결론 (한 줄 요약): 현재 버전에서 알림 팝업 노출 시간은 코드상 고정값으로 하드코딩되어 있어 관리자 정책/설정으로 조절할 수 없다. 이는 장애가 아닌 사양(기능 미제공)이며, 관리자페이지에 '알림 팝업 노출 시간(초)' 정책 항목을 추가하는 개선을 로드맵에 반영해 다음 마이너 버전을 목표로 지원 예정이다(SUP-144).

3.문의(요구사항) 요약

구분상세
요구 핵심 요청Mac 에이전트 알림 팝업이 화면에 떠 있는 시간(노출 시간)을 관리자가 조절할 수 있게 해달라.
확인 현재 가능 여부현재 버전에서는 조절 불가. 노출 시간이 코드 고정값으로 설정되어 있다.
미정 목표값고객이 원하는 구체적 초 단위 값·대상 OS(Mac/Windows)는 아직 미수집 → 후속 수집 필요(8장).
범위 대상 기능차단/안내 성격의 에이전트 팝업 UI. 모바일 푸시·결재 알림 등 타 알림 채널과는 무관.

4.로그 포렌식 분석 (장애 여부 확인)

이 문의가 장애인지 사양인지 먼저 판별하기 위해, 에이전트가 system.log에서 팝업/알림 관련 키워드(popup, notification)를 조회했다. 그 결과 관련 오류·이벤트 줄이 전혀 발견되지 않았다(0건). 즉, 시스템은 팝업 노출과 관련해 어떤 오류도 남기지 않았으며, 이는 기능이 정상 동작하고 있고 장애가 아님을 방증한다.

$ grep -i "popup" system.log 'popup' 를 포함하는 줄이 없습니다. $ grep -i "notification" system.log 'notification' 를 포함하는 줄이 없습니다. # 결과: 팝업/알림 관련 오류·이벤트 로그 0건 # → 장애의 흔적 없음. 노출 시간은 '기록되는 이벤트'가 아니라 # 코드 내부 고정 상수로 처리됨(사양).
Dead-end search(공회전 검색)의 의미: 통상 장애 분석에서 키워드 검색이 0건이면 조사가 막힌 것이지만, 이 케이스에서는 오히려 "오류가 없다 = 장애가 아니다"를 역으로 입증하는 근거가 된다. 노출 시간은 런타임에 로깅되는 값이 아니라 빌드에 박힌 고정 상수이므로, 로그에 흔적이 남지 않는 것이 정상이다.

검색 키워드대상결과해석

부수 관찰(무관): 팝업 키워드와 별개로, 같은 system.log에서 디스크 포화 및 감사로그 기록 실패 오류가 관측되었다. 이는 팝업 노출 시간 이슈와 전혀 무관하나 시급한 별건이므로 부록 B에 분리 기재한다.

5.현황 분석 — 조절 불가 원인

4장에서 장애가 아님이 확인되었으므로, 본 장은 근본 원인(RCA)이 아니라 "왜 현재 조절이 불가능한가"에 대한 현황 분석이다. 결론은 단순하다: 알림 팝업 노출 시간이 에이전트 코드상 고정값(상수)으로 하드코딩되어 있고, 관리자페이지 정책에 이를 제어할 설정 항목 자체가 존재하지 않기 때문이다.

[현황 — 조절 불가] 알림 팝업 노출 시간 = 코드 하드코딩 고정값 (예: 5초) | v 관리자페이지 정책에 해당 설정 항목 없음 | v 관리자가 값을 바꿀 방법이 없음 → ❌ 조절 불가 ────────────────────────────────────────────── [로드맵 — 조절 가능] 관리자페이지 정책에 '알림 팝업 노출 시간(초)' 항목 추가 | v 에이전트가 정책값을 fetch (정책 동기화) | v fetch 한 값으로 팝업 노출 시간 반영 → ✅ 조절 가능

현황 요인 정리

계층현황
표면 현상관리자가 알림 팝업 노출 시간을 변경할 UI/설정이 없다.
직접 원인노출 시간이 에이전트 코드의 고정 상수로 정의되어 있어 외부에서 주입할 경로가 없다.
구조적 배경관리자페이지 정책 스키마에 해당 파라미터가 미정의 → 에이전트가 참조할 정책값 자체가 없음.
결론장애가 아니라 기능 미제공(사양). 정책 항목 신설 + 에이전트 반영 개발이 필요.
핵심: 코드 버그를 고치는 문제가 아니라, "고정값을 정책 기반 설정값으로 승격"하는 기능 추가 문제다. 관리자페이지(정책 정의) + 에이전트(정책값 반영) 양쪽 작업이 함께 필요하다.

6.유사 사례 (Jira 매칭)

에이전트가 '팝업', '알림' 키워드로 사내 Jira를 조회했다. '팝업' 검색은 본 문의 이슈(SUP-144)를 정확히 회수했고, '알림' 검색은 3건이 매칭되었으나 그중 2건은 키워드만 겹칠 뿐 팝업 노출 시간과는 무관한 별개 이슈다.

SUP-144 조치 이력(resolution)

관리자페이지 정책에 '알림 팝업 노출 시간(초)' 설정 항목을 추가하고, 에이전트가 해당 정책값을 fetch·반영하도록 개선 예정. 로드맵에 반영되었으며 다음 마이너 버전을 목표로 한다. 임시 안내: 현재 버전은 고정값이라 조절 불가.

7.대응 방안 (현황 + 로드맵)

7-1. 현재 버전 — 조절 불가 (고정값)

현재 배포 버전에서는 노출 시간이 코드 고정값으로 박혀 있어 관리자·현장 모두 조절할 수 없다. 임시 우회책(로컬 설정 파일 편집 등)은 제공되지 않으며, 무리한 우회 대신 아래 정식 개선을 기다리는 것을 권장한다. 값이 반드시 즉시 필요한 경우에 한해, 개별 빌드로 상수를 조정한 버전을 제공할 수 있는지 개발팀 확인이 필요하다(8장).

7-2. 정식 대응 — 정책 기반 조절 (다음 마이너 버전)

관리자페이지 정책에 '알림 팝업 노출 시간(초)' 항목을 추가하고, 에이전트가 정책 동기화 시 이 값을 받아 팝업 노출 시간에 반영하도록 개선한다. 배포 후에는 관리자가 정책 화면에서 값을 지정하기만 하면 대상 단말에 적용된다.

계획된 관리자페이지 정책 설정 UI (목업)

┌─ 관리자페이지 · 정책 설정 › 알림 ────────────────────────┐ │ │ │ 알림 팝업 노출 시간 : [ 5 ] 초 │ │ (기본 5초, 범위 1 ~ 30초) │ │ │ │ 적용 대상 : [✓] Mac [ ] Windows │ │ │ │ [ 취소 ] [ 저장 ] │ └──────────────────────────────────────────────────────────┘ ※ 저장 시 정책 동기화를 통해 대상 에이전트에 반영됨.
동작 흐름: 관리자가 값 저장 → 정책 배포 → 에이전트가 다음 정책 fetch에서 값 수신 → 이후 노출되는 팝업부터 지정 시간 적용.

7-3. 개선 적용 체크리스트

  • 장애 여부 판별 완료 (로그 0건 → 사양 이슈로 확정)
  • 현황 원인 규명 (노출 시간 = 코드 하드코딩 고정값)
  • 관리자페이지 정책 스키마에 알림 팝업 노출 시간(초) 항목 추가 (기본 5초 · 범위 1~30초)
  • 에이전트가 정책값을 fetch하여 팝업 노출 시간에 반영하도록 개발
  • 적용 대상 OS(Mac/Windows) 분기 처리 검증
  • 다음 마이너 버전 릴리스에 포함 · 배포 후 고객사 D 적용 확인

8.후속 조치 & 커뮤니케이션 권고

  1. 고객사 안내: "현재 버전은 노출 시간이 고정값이라 조절 불가하며, 다음 마이너 버전에서 정책 기반 조절을 지원할 예정(SUP-144 로드맵 반영)"임을 안내한다.
  2. 요구사항 수집: 고객이 원하는 목표 노출 시간(초)과 대상 OS(Mac/Windows)를 확인해 SUP-144에 코멘트로 추가한다. (기본 5초 대비 얼마를 원하는지 구체값 확보)
  3. 일정 통지: 릴리스 일정이 확정되면 고객사 D에 통지한다. 급한 경우 상수만 조정한 개별 빌드 제공이 가능한지 개발팀에 확인한다.
  4. [별건·시급] 디스크 포화 조치: 로그 조사 중 발견된 /var/log 디스크 100% 포화 및 감사로그 기록 실패(503)는 팝업 이슈와 무관하나 즉시 조치가 필요한 별도 티켓으로 분리 처리한다(부록 B 참조).
주의 — 우선순위 역전: 정작 이번 문의(팝업 시간, Low)보다 부수 발견된 디스크 포화/감사로그 실패(별건)가 더 시급하다. 팝업 개선은 로드맵으로 관리하되, 디스크 이슈는 즉시 인프라팀에 에스컬레이션할 것.
