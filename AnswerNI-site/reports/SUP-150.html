1.개요 (Context)

고객사 G에서 본인확인(NICE 실명인증) 요청이 전건 실패한다는 문의가 접수되었다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 id-verify.log와 사내 Jira 이력을 조회·대조해 도출한 원인 분석과 조치 방안을 정리한 것이다. 조사 과정에서 오류 로그에 개인정보(PII)가 평문으로 적재되는 부가 문제도 확인되었다.

고객 문의 원문: "실명인증(본인확인)이 자꾸 실패한다는 문의예요. id-verify 로그 보고 원인 짚어줘"
결론 (한 줄 요약): 본인확인 제공사(NICE) API 키(apiKey)가 폐기·만료되어 토큰 갱신마저 401로 실패, 재발급 전까지 모든 verify 호출이 실패(code=E402)하는 것이 1차 원인. apiKey 재발급으로 복구하며, 오류 로그의 PII는 mask_pii_in_log=true로 마스킹해 재발 방지한다.

4.로그 포렌식 분석

근거 로그: logs/id-verify.log. 아래는 로컬 에이전트가 PII를 토큰으로 치환한 뒤 회수한 라인이다(원문 PII는 원격 LLM에 전달되지 않음 — 부록 B 참조).

2026-07-09 09:14:22 ERROR [idv] verify FAILED rid=IDV-20260709-0007 code=E402 reason="provider auth token expired" name=홍길동 user=[EMAIL_1] phone=[PHONE_1] rrn=[RRN_1] client_ip=[IP_1] 2026-07-09 09:15:48 ERROR [idv] verify FAILED rid=IDV-20260709-0012 code=E402 reason="provider auth token expired" name=김지원 user=[EMAIL_2] phone=[PHONE_2] rrn=[RRN_2] client_ip=[IP_2] 2026-07-09 09:18:31 ERROR [idv] verify FAILED rid=IDV-20260709-0021 code=E402 reason="provider auth token expired" name=이서준 user=[EMAIL_3] phone=[PHONE_3] rrn=[RRN_3] client_ip=[IP_3] 2026-07-09 09:20:10 WARN [idv] 34 verify requests failed in last 10m (provider auth token expired) — PII written to error log in cleartext 2026-07-09 09:22:36 ERROR [idv] provider token refresh FAILED — all verify calls will fail until apiKey is reissued
주의: 원본 로그에는 name/rrn/phone/email/client_ip가 평문으로 남아 있었다. AnswerNI 로컬 에이전트가 원격 LLM 전달 전 이를 [EMAIL_n]·[RRN_n]·[PHONE_n]·[IP_n] 토큰으로 치환했다(부록 B). 운영 로깅 자체도 마스킹으로 전환해야 한다.

5.근본 원인 분석 (RCA)

모든 verify 실패의 reason이 "provider auth token expired"(E402)로 동일하고, 토큰 자동 갱신마저 401 invalid_client: apiKey revoked로 실패한다. 즉 제공사 API 키가 폐기·만료되어 재발급 전까지 회복 불가능한 전건 실패 상태다.

[실명인증 요청] verify rid=IDV-... | v [제공사 토큰 검증] code=E402 provider auth token expired ❌ | v [토큰 자동 갱신 시도] http=401 invalid_client: apiKey revoked ❌ | v [verify 전건 실패] → 본인확인 불가 · 오류 로그에 PII 평문 적재

원인 사슬 (Causal Chain)

계층원인
직접 원인제공사 인증 토큰 만료(E402)로 verify 호출이 거부됨.
근본 원인제공사 API 키(apiKey)가 폐기·만료되어 토큰 갱신(refresh)마저 401로 실패 — 재발급 전까지 회복 불가.
기여 요인토큰 만료 사전 알림·자동 갱신 부재 · 오류 로그 PII 마스킹 미적용(mask_pii_in_log=false).
보안 관점: 실패 경로에서 주민번호·휴대폰·이메일·IP가 평문으로 로그에 적재된다. 로그 열람 권한자·로그 수집 파이프라인 전체로 개인정보가 확산될 수 있어, 기능 복구와 별개로 로깅 마스킹을 필수 조치로 처리해야 한다.

SUP-150 조치 이력(resolution)

제공사 API 키(apiKey)를 재발급·교체하고 토큰 갱신을 정상화하여 verify 복구. 부가 조치로 오류 로그의 PII를 마스킹(mask_pii_in_log=true)하도록 로깅 정책을 수정하고, 토큰 만료 30일 전 알림과 자동 갱신을 추가해 재발 방지. 조치 후 verify 정상, 로그 평문 PII 제거 확인.

7.해결 방안

7-1. 근본 해결 — 제공사 API 키 재발급 (권장)

폐기·만료된 제공사 apiKey를 새 키로 재발급·교체하고 토큰 갱신을 정상화한다. 교체 후 신규 verify 성공과 E402/token refresh 오류 소멸을 확인한다.

7-2. 필수 병행 — 오류 로그 PII 마스킹

로깅 정책을 mask_pii_in_log=true로 변경해 이름·주민번호·휴대폰·이메일·IP를 마스킹한다. 이미 남은 평문 PII 로그는 보관 정책에 따라 안전하게 정리한다.

# ❌ 현재 — 실패 시 요청 파라미터를 평문으로 기록
log.error(f"verify FAILED ... user={email} phone={phone} rrn={rrn} client_ip={ip}")

# ✅ 개선 — 마스킹 후 기록 (mask_pii_in_log=true)
log.error(f"verify FAILED ... user={mask(email)} phone={mask(phone)} "
          f"rrn={mask(rrn)} client_ip={mask(ip)}")

7-3. 배포 체크리스트

  • 근본 원인 규명 (apiKey 만료 → 토큰 갱신 401 → verify 전건 실패)
  • 제공사 apiKey 재발급·교체 후 verify 성공 확인
  • 로깅 정책 mask_pii_in_log=true 적용 + 기존 평문 PII 로그 정리
  • 토큰 만료 사전 알림(만료 30일 전) · 자동 갱신 구성
  • 배포 후 id-verify.log에서 E402/token refresh FAILED 미재발 확인

8.재발 방지 & 모니터링 권고

  • 토큰 수명 관리: 제공사 토큰 만료 30일 전 사전 알림 및 자동 갱신을 구성해 만료로 인한 전건 장애를 원천 차단한다.
  • 갱신 실패 모니터링: token refresh FAILED / http=401 발생 시 즉시 담당자에게 알림.
  • PII 로깅 표준화: mask_pii_in_log 정책을 전사 로그로 점검·표준화하여 다른 모듈에서도 평문 PII 적재가 없는지 확인한다.
  • 합성 모니터링: 실명인증 성공률 지표를 대시보드화해 전건 실패를 조기 감지한다.
  • 이슈 연계: 본 건을 SUP-150과 linked issue로 기록해 동일 증상 재문의 시 참조한다.

10.부록 A — 에이전트 분석 파이프라인

본 보고서는 로컬 데몬(도구 실행·PII 마스킹·SLM 요약)과 원격 LLM(Claude Code CLI, claude-opus-4-8)이 결합된 자율 tool-use 루프의 산출물이다. 누적 도구 호출 4회, 종료 사유 end_turn.

분석 특징: 자연어 문장 검색은 0건이었으나 핵심어(실명인증)로 축약 재검색하여 정답 이슈를 회수했다. 또한 로그 조회 즉시 로컬 에이전트가 PII를 치환해, 원문 개인정보가 원격 LLM에 전달되지 않았다.
