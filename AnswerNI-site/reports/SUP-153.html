1.개요 (Context)

고객사 J의 신규 지사(부산) PC들이 관리서버에 연결되지 않는다는 문의가 접수되었다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 network.log와 사내 Jira 이력을 조회·대조해 도출한 원인 분석과 조치 방안을 정리한 것이다.

고객 문의 원문: "신규 지사 PC들이 관리서버에 연결이 안 된대요. network 로그 보고 분석해줘"
결론 (한 줄 요약): 관리서버 포트(8443)로의 아웃바운드 통신이 고객사 방화벽에서 차단되어 에이전트 연결이 타임아웃(connection timed out)되는 것이 원인. 방화벽에 관리서버 IP:8443 아웃바운드 예외를 등록해 복구하며, 배포 전 필요 포트 목록을 사전 공유해 재발을 방지한다.

3.증상 요약

구분상세
증상 연결 타임아웃에이전트가 관리서버 198.51.100.8:8443 연결 시 connection timed out으로 반복 실패.
폴백 오프라인 캐시연결 실패 후 falling back to offline policy cache로 전환 — 정책 갱신 없이 캐시로만 동작.
결정적 경로 차단 지점traceroute mgmt: blocked after gateway 192.168.20.1 — 게이트웨이 이후 구간(방화벽)에서 경로가 끊김.
범위 대상 구간신규 지사(site=branch-busan) PC에 국한. 기존 구간은 정상 연결.

4.로그 포렌식 분석

근거 로그: logs/network.log. 에이전트 부팅 직후부터 관리서버 8443 연결이 타임아웃되고, 게이트웨이 이후에서 경로가 차단됨을 traceroute가 확인한다. 방화벽 예외 등록 이후 ok로 정상화된다.

2026-07-09 10:00:03 INFO [net] agent boot agent_id=PC-5502 site=branch-busan ip=192.168.20.44 2026-07-09 10:00:23 ERROR [net] connect to mgmt 198.51.100.8:8443 failed: connection timed out 2026-07-09 10:00:53 ERROR [net] connect to mgmt 198.51.100.8:8443 failed: connection timed out 2026-07-09 10:01:00 WARN [net] falling back to offline policy cache 2026-07-09 10:05:00 INFO [net] traceroute mgmt: blocked after gateway 192.168.20.1 2026-07-09 11:30:00 INFO [ops] firewall exception added for 198.51.100.8:8443 by netadmin=seo@corp.example.com 2026-07-09 11:32:10 INFO [net] connect to mgmt 198.51.100.8:8443 ok

주의: 로컬 게이트웨이(192.168.20.1)까지는 도달하나 그 이후에서 경로가 끊긴다. 이는 PC·에이전트 문제가 아니라 경계 방화벽의 아웃바운드 정책이 원인임을 가리킨다.

4-1. 지사 영향 확인 — 로그인 실패 정황 (auth-server.log)

관리 연결 차단 기간의 지사 측 영향을 파악하기 위해 에이전트가 auth-server.log를 교차 조회했다. 로그인 거부 라인에 이메일·IP·전화번호가 평문으로 남아 있어, 로컬 에이전트가 원격 LLM 전달 전 토큰으로 치환했다(원문 PII 미전달 — 부록 B 참조).

2026-07-08 09:12:45 WARN [auth] login rejected user=[EMAIL_1] ip=[IP_1] phone=[PHONE_1] 2026-07-08 09:13:02 WARN [auth] login rejected user=[EMAIL_1] ip=[IP_1] reason=cert_error retry=3

4-2. 오프라인 정책 미적용 (license.log)

관리 연결이 끊긴 동안 에이전트는 오프라인으로 동작하며 정책을 적용하지 못했다. 해당 경고 라인에도 사용자 이메일이 평문으로 포함되어 있어 마스킹 처리되었다.

2026-07-09 09:10:22 WARN [agent] policy not applied: license invalid agent_id=PC-4410 user=[EMAIL_2]
마스킹 안내: 위 두 로그(auth-server.log·license.log)에는 이메일·IP·전화번호 등 개인정보(PII)가 평문으로 존재했다. AnswerNI 로컬 에이전트가 원격 LLM 전달 전 이를 [EMAIL_n]·[IP_n]·[PHONE_n] 토큰으로 치환했다(부록 B). 주근거인 network.log 자체에는 PII가 없다.

5.근본 원인 분석 (RCA)

연결 실패가 특정 포트(8443)·특정 목적지(관리서버)에서만 발생하고, traceroute가 게이트웨이 이후 구간에서 차단을 보고한다. 로컬 네트워크(게이트웨이까지)는 정상이므로, 원인은 에이전트/PC가 아니라 고객사 방화벽 정책 변경으로 관리서버 8443 아웃바운드가 차단된 것이다.

[에이전트 PC-5502 · branch-busan] | v [로컬 게이트웨이 192.168.20.1] ✅ 도달 | v [경계 방화벽 — 아웃바운드 정책] ❌ 8443 차단 (blocked after gateway) | v [관리서버 198.51.100.8:8443] → connection timed out → offline policy cache 폴백

원인 사슬 (Causal Chain)

계층원인
직접 원인관리서버 8443 아웃바운드 연결이 타임아웃되어 에이전트가 관리서버에 접속하지 못함.
근본 원인고객사 방화벽 정책 변경으로 관리서버 IP:8443 아웃바운드가 차단됨(신규 지사 구간에 예외 미등록).
기여 요인신규 지사 배포 전 필요 포트 목록(8443 등) 사전 공유 부재 · 방화벽 화이트리스트 미반영.
운영 관점: 관리 연결이 끊긴 동안 에이전트는 오프라인 캐시로만 동작해 최신 정책·차단 규칙이 배포되지 않는다. 방화벽 예외 등록으로 관리 채널을 조속히 복구해야 정책 최신성이 유지된다.

6.유사 사례 (Jira 매칭)

에이전트가 로그에서 추출한 핵심어 '8443'으로 사내 Jira를 조회해 동일 증상의 과거 이슈 SUP-147을 회수했다(자연어 문장 검색은 0건 → 포트 번호 핵심어로 재검색해 매칭). 현재 문의는 본 이슈(SUP-153)로 등록되었다.

SUP-147 조치 이력(resolution)

방화벽에 관리서버 IP:8443 아웃바운드 예외를 등록하고, telnet/curl로 연결 확인 절차를 안내. 예외 등록 후 connect to mgmt ...:8443 ok 확인. 배포 전 필요한 포트 목록을 사전 공유하도록 프로세스 보완.

7.해결 방안

7-1. 근본 해결 — 방화벽 아웃바운드 예외 등록 (권장)

고객사 경계 방화벽에 관리서버 IP(198.51.100.8):8443 아웃바운드 예외를 등록한다. 신규 지사 구간(branch-busan)에서 관리서버로의 TCP 8443이 허용되도록 화이트리스트에 반영한다.

7-2. 연결 검증 — telnet/curl 확인

예외 적용 후 지사 PC에서 관리서버 포트 도달 여부를 직접 확인한다. connect ... ok 로그와 함께 정책 재동기화를 확인한다.

# 관리서버 8443 포트 도달 확인 (Windows)
Test-NetConnection 198.51.100.8 -Port 8443

# 또는 telnet / curl 로 확인
telnet 198.51.100.8 8443
curl -v --connect-timeout 5 https://198.51.100.8:8443/

7-3. 배포 체크리스트

  • 근본 원인 규명 (8443 아웃바운드 차단 → connection timed out → offline cache 폴백)
  • 방화벽에 관리서버 IP:8443 아웃바운드 예외 등록
  • 지사 PC에서 Test-NetConnection/telnet/curl로 8443 도달 확인
  • network.log에서 connect to mgmt ...:8443 ok 및 정책 재동기화 확인
  • 신규 지사 배포 표준에 필요 포트 목록(8443 등) 사전 공유 절차 반영

8.재발 방지 & 모니터링 권고

  • 배포 전 포트 사전 공유: 신규 지사·네트워크 구간 배포 시 관리서버 통신에 필요한 포트 목록(8443 등)을 고객사 네트워크 담당자에게 사전 공유해 방화벽 화이트리스트에 반영한다.
  • 연결 상태 모니터링: connection timed out / falling back to offline policy cache / blocked after gateway 패턴을 알림 규칙으로 등록해 관리 연결 단절을 조기 감지한다.
  • 온보딩 체크리스트: 신규 구간 온보딩 시 관리서버 포트 도달 확인(telnet/curl)을 배포 완료 조건으로 표준화한다.
  • 오프라인 캐시 가시화: 오프라인 폴백 상태가 지속되는 에이전트를 대시보드에 표시해 장기 미연결 PC를 추적한다.
  • 이슈 연계: 본 건(SUP-153)을 SUP-147과 linked issue로 기록해 동일 증상 재문의 시 참조한다.

10.부록 A — 에이전트 분석 파이프라인

본 보고서는 로컬 데몬(도구 실행·PII 마스킹·SLM 요약)과 원격 LLM(Claude Code CLI, claude-opus-4-8)이 결합된 자율 tool-use 루프의 산출물이다. 누적 도구 호출 6회, 종료 사유 end_turn.

11.부록 B — PII 마스킹 처리 내역

주근거인 network.log에는 PII가 없으나, 지사 영향 확인을 위해 교차 조회한 auth-server.log·license.log에 개인정보가 평문으로 포함되어 있었다. 로컬 에이전트가 원격 LLM 전달 전 이를 토큰으로 치환했으며(같은 값은 같은 토큰으로 매핑), 원문 PII는 원격 LLM에 전달되지 않는다.

토큰원문 유형출처 (로그 · 원문값)
[EMAIL_1]이메일auth-server.log · login rejected (kim@aaa.com)
[IP_1]클라이언트 IPauth-server.log · login rejected (203.0.113.42)
[PHONE_1]휴대폰auth-server.log · login rejected (010-1234-5678)
[EMAIL_2]이메일license.log · policy not applied (jung@corp.example.com)
참고: 주근거 로그 network.log 자체에는 마스킹 대상 PII가 없다(에이전트 식별자·사설 IP·포트·경로 정보만 포함). ops 라인의 netadmin 계정은 운영 담당자 식별용 정보로 개인정보 마스킹 대상에 해당하지 않는다.
