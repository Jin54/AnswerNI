1.개요 (Context)

고객사 H에서 다수 사용자가 로그인에 계속 실패하며 "SSL 때문인지 로그인이 안 된다"는 문의가 접수되었다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 auth-server.log와 사내 Jira 이력을 조회·대조하여 도출한 원인 분석과 조치 방안을 정리한 것이다.

고객 문의 원문: "SSL 때문인지 로그인이 계속 실패한다는 문의예요. auth-server 로그 보고 원인 정리해줘"
결론 (한 줄 요약): auth.corp.example.com의 서버 SSL 인증서가 2026-06-30 이후 갱신되지 않아 만료되어, TLS 핸드셰이크가 certificate has expired로 실패하고 로그인이 전건 거부되었다. 인증서 재발급·교체 후 auth-server 재시작으로 복구하며, 만료 30일 전 알림 자동화로 재발을 방지한다.

3.증상 요약

구분상세
증상 로그인 실패로그인 시도가 login rejected ... reason=cert_error로 거부. 사용자에게는 로그인 실패로 노출.
근원 인증서 만료SSL certificate verify failed: certificate has expired 및 certificate has expired (SSL_ERROR_EXPIRED_CERT)가 반복 기록.
조건 발생 시점2026-06-30 인증서 만료 이후 발생. 로그상 09:12:44에 첫 ERROR 관측.
범위 처리 경로auth-server의 TLS 핸드셰이크 단계. [auth]·[tls] 태그 로그에 집중, 정상 트래픽(health/pool)은 무관.

4.로그 포렌식 분석

주 근거 로그: demo/logs/auth-server.log. 08:00~09:12 구간은 정상 로그인·세션 검증이 이어지다가, 09:12:44 인증서 만료 ERROR를 기점으로 로그인 거부(WARN)가 연쇄한다. PII(이메일·IP·전화)는 로컬 에이전트가 원격 LLM 전달 전 토큰으로 치환했다.

4-1. auth-server.log — 인증서 만료 (주 근거)

2026-07-08 09:12:32 INFO [auth] token issued user=[EMAIL_2] ttl=3600s 2026-07-08 09:12:44 ERROR [auth] SSL certificate verify failed: certificate has expired 2026-07-08 09:12:45 WARN [auth] login rejected user=[EMAIL_1] ip=[IP_1] phone=[PHONE_1] 2026-07-08 09:12:46 ERROR [auth] SSL certificate expired for host auth.corp.example.com not renewed since 2026-06-30 2026-07-08 09:13:02 WARN [auth] login rejected user=[EMAIL_1] ip=[IP_1] reason=cert_error retry=3 2026-07-08 09:15:20 ERROR [tls] handshake failed: certificate has expired (SSL_ERROR_EXPIRED_CERT) 2026-07-08 13:05:41 ERROR [auth] login failure user=[EMAIL_1] phone=[PHONE_1] reason=ssl_certificate_expired 2026-07-08 15:10:04 ERROR [tls] certificate chain incomplete: unable to verify the first certificate

4-2. network.log — 인증 실패 시 오프라인 정책 캐시 폴백 (보조)

인증/관리 서버 접속이 실패하면 엔드포인트 에이전트가 정책을 갱신받지 못해 오프라인 정책 캐시로 폴백한다. network.log에서 관리 서버(mgmt) 접속이 연속 타임아웃한 뒤 폴백이 기록되었다 — 인증 계층 장애가 정책 배포까지 연쇄됨을 보여준다.

2026-07-09 10:00:23 ERROR [net] connect to mgmt 198.51.100.8:8443 failed: connection timed out 2026-07-09 10:00:53 ERROR [net] connect to mgmt 198.51.100.8:8443 failed: connection timed out 2026-07-09 10:01:00 WARN [net] falling back to offline policy cache
연관 해석: 인증서 만료로 TLS가 끊기면 에이전트-관리 서버 채널이 함께 막혀 정책이 최신화되지 않는다. 오프라인 캐시 폴백은 일시적 안전판일 뿐, 인증서 복구 없이는 정책 드리프트가 누적되므로 근본 조치(인증서 교체)가 선행되어야 한다.

5.근본 원인 분석 (RCA)

certificate has expired / SSL_ERROR_EXPIRED_CERT는 서버가 제시한 인증서의 유효기간(notAfter)이 지났을 때 TLS 라이브러리가 반환하는 전형적 오류다. 로그가 만료 호스트(auth.corp.example.com)와 미갱신 시점(2026-06-30)까지 명시하고 있어 원인은 명확하다.

[클라이언트 로그인 시도] TLS ClientHello | v [auth-server 인증서 제시] auth.corp.example.com (notAfter=2026-06-30) ❌ 이미 만료 | v [TLS 검증] → certificate has expired (SSL_ERROR_EXPIRED_CERT) | v [핸드셰이크 중단] → login rejected · reason=cert_error → 로그인 전건 실패

원인 사슬 (Causal Chain)

계층원인
직접 원인서버 인증서 만료로 TLS 핸드셰이크가 certificate has expired로 실패해 로그인이 거부됨.
근본 원인auth.corp.example.com 서버 인증서가 2026-06-30 이후 갱신되지 않음 (만료 갱신 누락).
기여 요인인증서 만료 사전 알림·모니터링 부재 — 만료 전 갱신 트리거가 없었음.
운영 관점: 인증서 만료는 시간이 지난다고 자연 회복되지 않으며, 수동 재발급·교체 후 서비스 재시작이 있어야만 복구된다. 15:10의 certificate chain incomplete는 교체 과정에서 중간 인증서(chain) 누락 위험도 함께 시사하므로, 재배포 시 풀체인(fullchain) 적용을 확인해야 한다.

6.유사 사례 (Jira 매칭)

에이전트가 로그에서 추출한 정확한 문구 'certificate expired'로 사내 Jira를 재검색해 동일 원인의 과거 이슈 SUP-123을 회수했다. 자연어 검색('로그인 실패')은 0건이었으나, 로그 근거로 키워드를 정제해 정답 이슈를 찾은 전형적 성공 패턴이다.

SUP-123 조치 이력(resolution)

만료된 서버 인증서를 재발급하여 교체하고 auth-server를 재시작해 해결. 조치: 1) 인증서 갱신(Let's Encrypt/사내 CA에서 재발급) 2) /etc/ssl/certs 경로에 신규 인증서 배포 3) systemctl restart auth-server 4) 만료 30일 전 알림 자동화(cron + monitoring) 추가로 재발 방지. 조치 후 로그인 정상화 확인.

7.해결 방안

7-1. 근본 해결 — 인증서 재발급·교체 (권장)

만료된 auth.corp.example.com 서버 인증서를 재발급하여 교체하고 auth-server를 재시작한다. 교체 시 중간 인증서를 포함한 풀체인(fullchain)을 배포해 chain incomplete 재발을 방지한다.

# 1) 인증서 재발급 (Let's Encrypt / 사내 CA)
certbot renew --cert-name auth.corp.example.com
# 또는 사내 CA에서 신규 인증서 발급

# 2) 신규 인증서(풀체인) 배포
cp fullchain.pem /etc/ssl/certs/auth.corp.example.com.pem
cp privkey.pem   /etc/ssl/private/auth.corp.example.com.key

# 3) 서비스 재시작 → 새 인증서 로드
systemctl restart auth-server

# 4) 만료일 검증
openssl x509 -enddate -noout -in /etc/ssl/certs/auth.corp.example.com.pem

7-2. 재발 방지 — 만료 30일 전 알림 자동화

인증서 notAfter를 주기적으로 점검해 만료 30일 전 알림을 발송하는 자동화를 추가한다(cron + 모니터링). SUP-123에서도 동일 조치로 재발을 막았다.

7-3. 조치 체크리스트

  • 근본 원인 규명 (서버 인증서 만료 → certificate has expired)
  • 만료 인증서 재발급 및 풀체인 교체
  • systemctl restart auth-server로 새 인증서 로드
  • 교체 후 로그인·TLS 핸드셰이크 정상 동작 검증
  • 만료 30일 전 알림 자동화(cron + monitoring) 등록

8.재발 방지 & 모니터링 권고

  • 만료 알림 자동화: 전 서비스 인증서의 notAfter를 스캔해 만료 30일/7일 전 알림을 발송한다.
  • 로그 알림 규칙: certificate has expired / SSL_ERROR_EXPIRED_CERT / login rejected ... reason=cert_error 패턴을 알림으로 등록해 조기 감지한다.
  • 풀체인 배포 표준화: 인증서 교체 시 중간 인증서 포함 여부를 배포 파이프라인에서 검증해 chain incomplete를 예방한다.
  • 인증 로그 PII 최소화: 로그인 실패 로그의 이메일·IP·전화 평문 적재를 마스킹하거나 최소 수집으로 전환한다.
  • 이슈 연계: 본 건을 SUP-123과 linked issue로 기록해 동일 원인 재문의 시 참조한다.

10.부록 A — 에이전트 분석 파이프라인

본 보고서는 로컬 데몬(도구 실행·PII 마스킹·요약)과 원격 LLM(Claude Code CLI, claude-opus-4-8)이 결합된 자율 tool-use 루프의 산출물이다. 누적 도구 호출 5회, 종료 사유 end_turn.

도구 1 · 로그 조회
auth-server.log (키워드 ERROR) → SSL 인증서 만료·TLS 핸드셰이크 실패 확인
도구 2 · 로그 조회
auth-server.log (키워드 certificate) → 만료 호스트·미갱신 시점(2026-06-30) 특정
도구 3 · 로그 조회
network.log (키워드 offline) → 인증 실패 연쇄로 falling back to offline policy cache 확인
도구 4 · Jira 검색
'로그인 실패' → 매치 0건 (자연어로는 미회수)
도구 5 · Jira 검색
'certificate expired' → 매치 1건 (SUP-123 회수)
종료
보고서 생성 완료 (stop_reason=end_turn, 누적 도구 호출 5회)
분석 특징: 첫 Jira 검색(자연어 '로그인 실패')은 0건이었으나, 로그에서 추출한 정확한 예외 문구 'certificate expired'로 재검색하여 정답 이슈(SUP-123)를 회수했다 — 로그 근거 → 키워드 정제 → 이력 매칭의 전형적 성공 패턴.

11.부록 B — PII 마스킹 처리 내역

로컬 에이전트가 원격 LLM 전달 전 로그 내 개인정보를 토큰으로 치환했다. 원문 PII는 원격 LLM에 전달되지 않는다. 아래는 auth-server.log의 로그인 거부(WARN)·실패(ERROR) 라인에서 치환된 항목이다.

토큰원문 유형출처 (원문값)
[EMAIL_1]로그인 거부 사용자 이메일09:12:45 / 09:13:02 WARN, 13:05:41 ERROR (kim@aaa.com)
[IP_1]클라이언트 IP09:12:45 / 09:13:02 WARN (203.0.113.42)
[PHONE_1]휴대폰 번호09:12:45 WARN, 13:05:41 ERROR (010-1234-5678)
참고: 인증서 만료 ERROR 라인(certificate has expired 등)에는 PII가 포함되지 않아 마스킹 없이 원문 그대로 원격 LLM에 전달되었다. 원인 분석에 필요한 기술 정보는 보존하면서 개인정보만 선택적으로 치환된다.
