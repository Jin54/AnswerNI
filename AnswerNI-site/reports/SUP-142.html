1.개요 (Context)

고객사 E의 운영 담당자가 "특정 시간대에 관리서버 응답이 느려진다"는 문의를 접수했다. 별도의 명시적 오류 메시지 없이 화면 로딩·API 응답이 지연되는 형태였으며, 담당자는 정확한 원인을 특정하지 못한 채 로그 분석을 요청했다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 logs/webhook.log를 비롯한 다중 로그와 사내 Jira 이력을 조회·대조하여 도출한 원인 분석과 조치 방안을 정리한 것이다.

고객 문의 원문: "특정 시간에 관리서버가 느려진다는 문의가 있어요. 로그 보고 원인 짚어줘"
결론 (한 줄 요약): 비정상 설치된 에이전트(agent_id=null)가 service-assign webhook을 호출하며 최대 3회 재시도(최악 ~13초) 동안 DB 커넥션을 오래 점유 → 동시간대 HikariCP 커넥션 풀 고갈(active=20 / idle=0 / waiting=14) → 후속 요청 30초 타임아웃 → 평균 응답 8.7초로 degraded. webhook 재시도 max-attempts를 3→1로 축소하고 connect/read 타임아웃을 3초로 설정, 실패 로깅을 강화하여 근본 차단한다.

5.근본 원인 분석 (RCA)

비정상 설치로 agent_id=null 상태인 에이전트가 service-assign webhook을 호출한다. 서버는 agent id를 찾지 못해 최대 3회까지 재시도(각 시도당 약 4초, 최악 누적 ~13초)하는데, 이 재시도 구간 내내 DB 커넥션을 점유한다. 동시간대에 이런 실패 webhook이 반복되면 커넥션이 반환되지 못하고 쌓여 풀이 고갈된다.

[비정상 설치 에이전트] service-assign webhook 호출 (agent_id=null) | v [재시도 루프] attempt 1/3 (~4s) → 2/3 (~4s) → 3/3 (~4s) → fail (~13s) ❌ 취약 지점 | └─ 재시도 구간 내내 DB 커넥션 점유 (반환 지연) v [HikariCP 커넥션 풀] active=20 · idle=0 · waiting=14 → pool exhausted | v [신규 요청] 커넥션 획득 대기 → request timed out after 30000ms | v [관리서버] avg response 8.7s → degraded (전 사용자 체감 지연)

원인 사슬 (Causal Chain)

계층원인
직접 원인webhook 재시도(최대 3회, 최악 ~13초) 동안 DB 커넥션을 장시간 점유하여 HikariPool이 고갈(active=20 idle=0 waiting=14)됨.
근본 원인비정상 설치(agent_id=null) 에이전트가 유발하는 webhook 재시도 폭주(retry storm) + 과도한 재시도 횟수·타임아웃 설정(3회 / 사실상 장시간 대기).
기여 요인실패 webhook에 serial·agent 식별자 로깅 부재로 원인 특정 지연 · 커넥션 풀 크기/타임아웃 방어선 부재 · 비정상 설치를 걸러내는 등록 검증 부재.
구조적 관점: 하나의 느린 외부 호출(webhook)이 공유 자원(커넥션 풀)을 붙잡으면, 그 여파는 해당 기능을 넘어 시스템 전체로 전파된다. 따라서 "재시도를 줄이는 것"만이 아니라 느린 의존성이 공유 자원을 오래 붙잡지 못하도록 타임아웃·서킷 브레이커로 경계를 두는 것이 근본 방어다.

6.유사 사례 (Jira 매칭)

에이전트가 로그에서 추출한 키워드(connection pool exhausted / webhook 재시도 / row lock / No space left)로 사내 Jira를 조회한 결과, 동일 증상의 SUP-142가 완전 일치(primary)로 회수되었다. "느림" 조사 과정에서 표면화된 복합 정황과 연관된 SUP-148·SUP-145는 참고(복합 정황)로 함께 제시한다.

webhook 재시도 max-attempts를 3→1로 축소(최악 13초 → 3초)하고, connect/read 타임아웃을 3초로 설정. 실패 시 serial·agent 식별자를 포함한 상세 에러 로깅을 추가하여 원인 추적성을 확보. 적용 후 평균 응답시간 정상화를 확인했다.

참고 이슈 취급: SUP-148(락 대기)·SUP-145(디스크 포화)는 에이전트가 '느림'을 조사하며 복합 정황 로그로부터 매칭한 항목이다. 본 건의 직접 원인과는 별개이므로 후속 시스템 건전성 점검 항목으로만 연계 기록한다.

7.해결 방안

7-1. 근본 해결 — 재시도 축소 & 타임아웃 경계 (권장)

webhook 재시도 횟수를 줄여 실패가 빨리 확정되도록 하고, connect/read 타임아웃을 짧게(3초) 설정하여 느린 호출이 커넥션을 오래 붙잡지 못하도록 경계를 둔다. 이로써 최악 소요가 ~13초에서 ~3초로 단축되어 풀 점유 시간이 대폭 감소한다.

# webhook 클라이언트 설정 (before → after)
webhook:
  retry:
    maxAttempts: 3   → 1          # 재시도 폭주 차단 (최악 13s → 3s)
  timeout:
    connectTimeout: 30s → 3s   # 연결 대기 상한
    readTimeout:    30s → 3s   # 응답 대기 상한
  onFailure:
    log: "serial + agent_id 포함 상세 에러 로깅" // 신규

7-2. 원천 차단 — 비정상 설치(agent_id=null) 방지

실패 webhook의 대상이 모두 agent_id=null인 점에서, 근본적으로는 agent id 없이 등록되는 비정상 설치를 원천 차단해야 한다. 설치/등록 절차에 식별자 검증(유효한 agent_id 없으면 등록 거부)을 강화하여 무의미한 재시도 호출 자체를 발생시키지 않는다.

7-3. 부가 — 자원 방어선 재점검

  • 커넥션 풀 재점검: HikariCP maximumPoolSize·connectionTimeout·maxLifetime을 부하 특성에 맞게 재조정하고, 풀 고갈 시 빠른 실패(fail-fast) 정책을 검토한다.
  • 서킷 브레이커 / 백오프: webhook 등 외부 의존 호출에 서킷 브레이커와 지수 백오프를 도입해 반복 실패가 자원을 잠식하지 않도록 한다.

7-4. 배포 체크리스트

  • 근본 원인 규명 (webhook 재시도 폭주 → 커넥션 풀 고갈 → 응답 지연)
  • webhook max-attempts 3→1 축소 적용
  • connectTimeout / readTimeout 3초 설정
  • 실패 시 serial·agent_id 포함 상세 에러 로깅 추가
  • 비정상 설치(agent_id=null) 등록 검증 강화
  • 커넥션 풀 크기·타임아웃 재점검 및 서킷 브레이커/백오프 검토
  • 배포 후 webhook.log/db.log에서 pool exhausted·timed out 미재발 및 평균 응답 정상화 확인

8.재발 방지 & 모니터링 권고

  • 커넥션 풀 상시 모니터링: HikariPool의 active/idle/waiting 지표를 지속 수집하고, waiting > 임계치 또는 idle=0 지속 시 알림 규칙을 등록해 고갈을 조기 감지한다.
  • webhook 실패율 알림: webhook failed after N attempts 및 elapsed 지표를 집계하여 실패 급증 시 자동 통지한다.
  • 응답 지연 알림: avg response time의 degraded 임계(예: 3s↑)를 SLO로 관리하고 위반 시 온콜 통지한다.
  • 비정상 설치 검출: agent_id=null webhook 호출을 별도 카운터로 추적하여 비정상 설치 유입을 조기에 차단·회수한다.
  • 복합 정황 후속 점검: 참고로 관측된 메모리 증가(agent-perf.log)·디스크 포화(system.log)·락 대기(db.log)는 별도 티켓(SUP-148/SUP-145 연계)으로 추적한다.
  • 이슈 연계: 본 건을 SUP-142와 linked issue로 기록하고, 동일 "느림" 재문의 시 커넥션 풀·webhook 재시도를 1차 점검 포인트로 참조한다.
