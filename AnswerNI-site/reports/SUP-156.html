1.개요 (Context)

고객사 M에서 로그인·세션·계정·본인확인 전반에 걸쳐 여러 증상이 겹쳐 산발적으로 발생한다는 문의가 접수되었다. 단일 결함이 아니라 인증 스택 전반의 정황이 뒤섞인 광역 이슈로, 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 auth-server / db / license 3종 로그와 사내 Jira 이력을 광역 검색으로 조회·대조해 도출한 종합 대응 가이드다.

고객 문의 원문: "요즘 로그인·인증 쪽에서 여러 증상이 겹쳐 발생해요. 과거 유사 사례들을 폭넓게 찾아서 정리해줘"
결론 (한 줄 요약): 증상이 인증서 만료(로그인 거부)·세션 TTL·재설정 메일·본인확인·라이선스 만료(정책 미적용)·DB 락(결재 저장 실패) 등 여러 영역에 걸쳐 있어, 광역 검색으로 유사 사례 8건을 회수하고 영역별 조치 이력을 종합해 점검 체크리스트로 정리했다. 단일 근본원인이 아니므로 영역별 병렬 점검이 핵심.

4.로그 포렌식 분석 (다중 로그)

광역 증상 특성상 단일 로그로 좁히지 않고 auth-server · db · license 3종 로그를 순차 조사했다. 아래는 로컬 에이전트가 PII를 토큰으로 치환한 뒤 회수한 라인이다(원문 PII는 원격 LLM에 전달되지 않음 — 부록 B 참조).

4-1. auth-server.log — 인증서 만료로 로그인 거부

2026-07-08 09:12:44 ERROR [auth] SSL certificate verify failed: certificate has expired 2026-07-08 09:12:45 WARN [auth] login rejected user=[EMAIL_1] ip=[IP_1] phone=[PHONE_1] 2026-07-08 09:12:46 ERROR [auth] SSL certificate expired for host auth.corp.example.com not renewed since 2026-06-30 2026-07-08 09:13:02 WARN [auth] login rejected user=[EMAIL_1] ip=[IP_1] reason=cert_error retry=3 2026-07-08 09:15:20 ERROR [tls] handshake failed: certificate has expired (SSL_ERROR_EXPIRED_CERT)

4-2. db.log — 행 잠금 대기로 결재 저장 실패

2026-07-08 16:40:31 WARN [db] slow query 30021ms UPDATE policy SET ... (T1 holds row lock) 2026-07-08 16:40:35 ERROR [db] tx T2 lock wait timeout exceeded; try restarting transaction 2026-07-08 16:40:35 ERROR [adm] approve save failed doc=3120 approver=[EMAIL_2] reason=lock_wait_timeout 2026-07-08 16:41:10 WARN [db] 6 transactions waiting on row lock (table=approve) 2026-07-08 16:45:01 INFO [db] lock contention cleared

4-3. license.log — 라이선스 만료로 정책 미적용

주의: 세 로그 모두 실패/거부 라인에 이메일·IP·휴대폰·결재자 이메일이 평문으로 남아 있었다. AnswerNI 로컬 에이전트가 원격 LLM 전달 전 [EMAIL_n]·[IP_n]·[PHONE_n] 토큰으로 치환했다(부록 B). 운영 로깅 자체도 마스킹으로 전환해야 한다.

5.근본 원인 분석 (RCA)

본 건은 단일 근본원인이 아니라 인증 스택 전반의 서로 다른 영역이 동시에 열화된 광역 이슈다. 로그 3종에서 확인된 인증서 만료 · DB 락 · 라이선스 만료가 각각 독립적으로 로그인/결재/정책 경로를 흔들고 있으며, 여기에 세션 TTL·본인확인·재설정 메일·푸시 등 인접 증상이 겹쳐 "여러 증상이 동시에" 체감된다.

[고객 체감] "로그인·인증 전반이 이상하다" | +── auth-server.log ──▶ 인증서 만료(2026-06-30 이후 미갱신) → 로그인 rejected | +── db.log ───────────▶ 장기 트랜잭션 행 잠금 → 결재 저장 lock_wait_timeout | +── license.log ──────▶ 라이선스 만료 → safe mode, 정책 미적용 | v [종합] 단일 원인 아님 → 영역별 병렬 점검 + 과거 유사 이력 8건 대조

6.유사 사례 (Jira 매칭 · 8건)

에이전트가 자연어 문장 대신 광역 키워드('로그인' → '인증' → '계정 인증')로 범위를 넓혀 재검색하여 최종적으로 동일·인접 증상의 과거 이슈 8건을 회수했다(현재 문의 SUP-156 포함). 아래 8건을 영역별로 묶어 종합 대응의 근거로 삼는다.

7.종합 대응 — 영역별 점검 체크리스트

단일 조치로 해소되지 않는 광역 이슈이므로, 회수한 8건의 과거 조치를 영역별로 이식한 병렬 점검 체크리스트를 제시한다.

7-1. 인증서 (SUP-123)

  • auth-server.log의 certificate has expired / not renewed since 2026-06-30 확인
  • 서버 인증서(auth.corp.example.com) 재발급·교체 후 auth-server 재시작
  • 인증서 만료 30일 전 알림 자동화(cron + monitoring) 등록

7-2. 세션 TTL (SUP-101)

  • 세션 TTL 설정값 점검(오설정 시 3600초로 복원) · 배포 환경변수 오타 확인

7-3. 재설정 메일 (SUP-108)

  • 발신 도메인 SPF/DKIM 레코드 존재 확인 · 스팸 분류 여부 점검

7-4. 본인확인 (SUP-150)

  • 제공사 apiKey/토큰 유효성 확인, 만료 시 재발급 · 오류 로그 PII 마스킹(mask_pii_in_log=true)

7-5. 라이선스 (SUP-146)

  • license.log의 license expired -> safe mode / policy not applied 확인
  • 라이선스 갱신·재적용 후 정책 재동기화(policy re-sync ok) 확인

7-6. 푸시 (SUP-131)

  • APNs 인증 키(.p8) 만료 여부 확인, 만료 시 신규 키 발급

7-7. DB 락 (SUP-148)

  • db.log의 lock wait timeout exceeded / approve save failed 확인
  • 장기 실행 쿼리 튜닝·트랜잭션 분할, 락/커넥션 풀 타임아웃 조정, 슬로우 쿼리 모니터링 추가

8.재발 방지 & 모니터링 권고

  • 만료 관리 통합: 인증서·라이선스·제공사 토큰·APNs 키 등 만료성 자산의 만료 30일 전 알림을 하나의 대시보드로 통합 관리한다(SUP-123·146·150·131 공통 교훈).
  • 락 모니터링: lock wait timeout / transactions waiting on row lock 패턴을 알림 규칙으로 등록(SUP-148).
  • PII 로깅 표준화: auth·db·license 등 전 로그의 mask_pii_in_log 정책을 점검·표준화해 평문 PII 적재를 제거한다(SUP-150).
  • 광역 증상 트리아지: "여러 증상이 겹친다"는 문의는 단일 원인으로 좁히지 말고 영역별 로그 병렬 조회 + 광역 Jira 검색으로 분리 진단한다.
  • 이슈 연계: 본 건을 SUP-123·101·108·150·146·131·148과 linked issue로 기록해 재문의 시 참조한다.
