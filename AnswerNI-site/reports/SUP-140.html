1.개요 (Context)

고객사 A의 관리자가 DLP 결재(승인) 화면에서 반려/승인 의견을 저장할 때 'Internal Server Error'(HTTP 500)가 발생하여 의견이 저장되지 않는다고 문의했다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 관련 로그와 사내 Jira 이력을 조회·대조하여 도출한 원인 분석과 조치 방안을 정리한 것이다.

고객 문의 원문: "결재 의견 저장 시 Internal Server Error가 난다는 문의예요. 원인과 해결책 정리해줘"
결론 (한 줄 요약): 의견 텍스트에 포함된 작은따옴표(')가 SQL 문자열 경계로 오인되어 쿼리가 파손(unterminated quoted string)되는 것이 원인. Prepared Statement(파라미터 바인딩) 적용으로 근본 차단하며, 배포 전까지는 '→'' 이스케이프로 즉시 완화한다.

구분상세

5.근본 원인 분석 (RCA)

unterminated quoted string 예외는 사용자 입력값을 문자열 연결(string concatenation) 방식으로 SQL에 직접 삽입할 때 나타나는 전형적 증상이다. 입력에 포함된 '가 문자열 리터럴의 종료 경계로 오인되어 쿼리 구문이 붕괴한다.

[결재 의견 입력] "반려합니다 'A안'은..." | v [쿼리 조립 — 문자열 연결] ❌ 취약 지점 INSERT INTO opinion(...) VALUES('반려합니다 'A안'은...') └──┬──┘└──────────── ' 가 문자열을 조기 종료 | v [PostgreSQL 파서] → unterminated quoted string at or near "'" | v [예외 → HTTP 500] → 의견 저장 실패 · 사용자에게 오류 알림

원인 사슬 (Causal Chain)

계층원인
직접 원인의견 문자열의 '가 SQL 문자열 경계로 해석되어 쿼리 구문 파손.
근본 원인입력값을 바인딩 파라미터가 아닌 문자열 연결로 쿼리를 구성하는 저장 로직.
기여 요인특수문자 입력에 대한 회귀 테스트 부재 · 입력 검증/이스케이프 계층 없음.
보안 관점: 동일한 결함은 악의적 입력 시 SQL Injection으로 악용될 수 있다. 우발적 500 오류와 인젝션 취약점은 같은 뿌리(문자열 연결 쿼리)를 공유하므로, 단순 이스케이프가 아닌 파라미터 바인딩으로 근본 차단해야 한다.

에이전트가 'unterminated quoted string' 키워드로 사내 Jira를 조회해 동일 증상의 과거 이슈 SUP-140을 회수했다(현재 문의가 곧 이 이슈의 재현).

SUP-140 조치 이력(resolution)

입력 의견의 작은따옴표를 이스케이프('→'')하여 저장하도록 수정하고, 서버측 파라미터 바인딩(Prepared Statement)을 병행 적용해 근본 차단. 재현 조건: 의견 텍스트에 ' 1개 이상 포함. 수정 후 정상 저장 확인.

7.해결 방안

7-1. 근본 해결 — 파라미터 바인딩 (권장)

의견 저장 쿼리에서 입력값을 문자열로 직접 연결하지 말고 바인딩 파라미터(placeholder)로 전달하도록 수정한다. ' 등 특수문자가 데이터로만 처리되어 쿼리 파손·SQL 인젝션이 원천 차단된다.

// ❌ 취약 — 문자열 연결
"INSERT INTO opinion(doc_id, body) VALUES('" + docId + "', '" + opinion + "')"

// ✅ 안전 — Prepared Statement (파라미터 바인딩)
PreparedStatement ps = conn.prepareStatement(
  "INSERT INTO opinion(doc_id, body) VALUES(?, ?)");
ps.setLong(1, docId);
ps.setString(2, opinion);  // ' 가 데이터로만 처리됨
ps.executeUpdate();

7-2. 즉시 완화 — 작은따옴표 이스케이프

코드 수정 배포 전 임시 대응으로, 저장 직전 의견 문자열의 '를 ''로 이스케이프한다. (SUP-140에서도 병행 적용) 단, 이는 한시적 대응이며 근본 해결(7-1)로 대체되어야 한다.

7-3. 배포 체크리스트

  • 근본 원인 규명 (문자열 연결 쿼리 → unterminated quoted string)
  • 결재 의견 저장 쿼리를 Prepared Statement로 전환
  • 임시 완화('→'')를 배포 전까지 적용
  • 회귀 테스트: ', ", 개행, 유니코드 특수문자 포함 케이스 저장 검증
  • 배포 후 admin-approve.log에서 500 / unterminated quoted string 미재발 확인

8.재발 방지 & 모니터링 권고

  • 코드 전수 점검: 결재/의견 저장 외에도 사용자 입력을 쿼리에 직접 연결하는 다른 API 경로가 있는지 점검하여 동일 패턴의 잠재 결함 및 SQL 인젝션 위험을 제거한다.
  • 회귀 테스트 추가: ', ", 개행 등 특수문자 입력에 대한 저장 테스트 케이스를 CI에 편입한다.
  • 모니터링: admin-approve.log의 unterminated quoted string / 500 / N approval-save failures 패턴을 알림 규칙으로 등록해 조기 감지한다.
  • DEBUG 로깅 정리: opinion_raw 평문 + PII 동시 로깅은 운영 환경에서 비활성화하거나 마스킹한다.
  • 이슈 연계: 본 건을 SUP-140과 linked issue로 기록해 동일 증상 재문의 시 참조한다.
