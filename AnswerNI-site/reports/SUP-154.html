1.개요 (Context)

고객사 K에서 특정 일자부터 에이전트의 검사·차단이 동작하지 않고 정책도 적용되지 않는다는 문의가 접수되었다. 본 보고서는 AnswerNI 자율 에이전트 파이프라인(app.agent.run_agent)이 license.log를 주근거로 admin-approve.log·dlp-approve-api.log 정황과 사내 Jira 이력을 교차 조회·대조하여 도출한 원인 분석과 조치 방안을 정리한 것이다.

고객 문의 원문: "오늘부터 검사·차단이 동작을 안 하고 정책도 안 먹어요. 왜죠?"
결론 (한 줄 요약): 제품 라이선스가 만료되어 에이전트가 안전모드(safe mode)로 전환, 검사·차단·정책 적용이 일괄 비활성화된 것이 원인(license expired -> entering safe mode (block/scan disabled)). 라이선스 갱신·재적용 + 정책 재동기화로 복구하며, 재발 방지를 위해 만료 30일 전 자동 알림을 권고한다.

3.증상 요약

구분상세
증상 검사·차단 정지에이전트가 안전모드로 전환되어 검사(scan)·차단(block)이 동작하지 않음(block/scan disabled).
증상 정책 미적용정책 배포가 반영되지 않음 — policy not applied: license invalid.
전조 만료 임박 경고당일 license expires today WARN이 선행 기록됨.
범위 처리 경로라이선스로 게이팅되는 에이전트 전 기능(scan/block/policy)에 광범위 영향.

4.로그 포렌식 분석

에이전트는 3개 로그를 교차 조회했다. license.log가 결정적 주근거이며, admin-approve.log·dlp-approve-api.log는 안전모드 기간에도 결재·승인 흐름이 통제 없이 계속 시도되는 정황을 뒷받침한다. 아래 로그는 로컬 에이전트가 PII를 토큰으로 치환한 뒤 회수한 것이다(원문 PII는 원격 LLM에 전달되지 않음 — 부록 B 참조).

4-1. license.log — 결정적 주근거

2026-07-09 00:00:05 INFO [lic] license check ok expire=2026-07-08 2026-07-09 00:00:05 WARN [lic] license expires today 2026-07-09 00:05:00 ERROR [lic] license expired -> entering safe mode (block/scan disabled) 2026-07-09 09:10:22 WARN [agent] policy not applied: license invalid agent_id=PC-4410 user=[EMAIL_1] 2026-07-09 09:40:15 WARN [agent] scan skipped: license invalid 2026-07-09 13:20:00 INFO [lic] license renewed expire=2027-07-08 by=[EMAIL_5] 2026-07-09 13:21:00 INFO [agent] policy re-sync ok

00:05 만료와 동시에 안전모드 진입(block/scan disabled), 이후 policy not applied·scan skipped가 이어진다. 13:20 라이선스 갱신 직후 policy re-sync ok로 정상화되는 인과의 시작–끝이 한 로그에 모두 담겨 있다.

4-2. admin-approve.log — 정책·결재 흐름 영향 정황

2026-07-09 14:18:02 INFO [adm] approve console opened approver=[EMAIL_2] ip=[IP_1] 2026-07-09 14:21:03 DEBUG [adm] opinion_raw="반려합니다 'A안'은 재검토 필요" approver=[EMAIL_3] contact=[PHONE_1] 2026-07-09 14:25:12 DEBUG [adm] opinion_raw="'긴급' 승인 요청 반려" approver=[EMAIL_4]

안전모드 구간에도 관리자는 결재 콘솔을 열고 승인·반려 작업을 계속한다. 그러나 정책 적용이 무효화된 상태이므로 이 흐름은 DLP 검사·차단 통제 없이 진행된다. 아울러 결재자 이메일·연락처가 평문으로 로깅되어 로컬 에이전트가 [EMAIL_2]·[EMAIL_3]·[EMAIL_4]·[PHONE_1]·[IP_1]로 치환했다.

4-3. dlp-approve-api.log — 클라이언트 승인 경로 정황

2026-07-08 15:01:48 INFO "POST /client/v1/dlp/approve/CUST9931/2438/fileattach" 200 81 2026-07-08 15:02:33 INFO [adm] approve changed doc=2438 by=[EMAIL_5] ip=[IP_2] result=완료

클라이언트 DLP 승인/검사 요청은 200으로 처리되지만, 안전모드에서는 검사·차단 enforcement가 실제로 걸리지 않아 승인 경로가 통제 공백 상태로 흐른다. 이 경로에서도 관리자 이메일·IP가 평문으로 남아 [EMAIL_5]·[IP_2]로 마스킹 처리했다.

5.근본 원인 분석 (RCA)

모든 증상의 기점은 라이선스 만료 → 안전모드 전환 단 하나의 사건이다. 안전모드는 라이선스가 유효하지 않을 때 제품이 취하는 보호 상태로, 이 상태에서는 검사(scan)·차단(block)·정책(policy) 적용이 일괄 중단된다. 따라서 개별 기능의 결함이 아니라 라이선스 게이팅에 의한 일괄 비활성화가 근본 원인이다.

[00:00 license expires today] ← 만료 임박 경고(선행 신호) | v [00:05 license expired] → entering safe mode (block/scan disabled) ❌ | v [09:10 policy not applied: license invalid] · [09:40 scan skipped: license invalid] | v [검사·차단·정책 전면 정지] → 통제 공백 · 결재/승인 흐름만 무방비로 지속 | v [13:20 license renewed] → [13:21 policy re-sync ok] ✅ 정상화

원인 사슬 (Causal Chain)

계층원인
직접 원인라이선스 만료로 에이전트가 안전모드로 전환되어 block/scan disabled 상태가 됨.
근본 원인라이선스가 갱신되지 않은 채 만료일(2026-07-08)을 경과 — 유효 라이선스 부재로 모든 게이팅 기능이 무효화.
기여 요인만료 사전 알림·자동 갱신 부재 · 안전모드 진입에 대한 경보 미설정 · 로그 PII 평문 적재.
보안 관점: 안전모드는 제품을 "고장내지 않기 위한" 안전장치이지만, 그 대가로 DLP 통제가 전면 공백이 된다. 만료~갱신 사이 구간은 유출 탐지·차단이 이뤄지지 않으므로, 라이선스 수명 관리를 운영 리스크로 상시 관리해야 한다.

SUP-146 조치 이력(resolution)

라이선스를 갱신·재적용하고 정책을 재동기화하여 정상화. 재발 방지를 위해 만료 30일 전 자동 알림 발송을 권고. 조치 후 policy re-sync ok 확인.

7.해결 방안

7-1. 근본 해결 — 라이선스 갱신·재적용 + 정책 재동기화 (권장)

만료된 라이선스를 갱신·재적용하면 에이전트가 안전모드를 해제하고 검사·차단·정책이 복구된다. 갱신 직후 정책 재동기화(re-sync)를 수행해 만료 기간 동안 반영되지 못한 정책을 일괄 배포한다. 로그상 license renewed → policy re-sync ok 순으로 정상화가 확인된다.

[관리 콘솔] 라이선스 > 갱신 1) 신규 라이선스 키 등록 / 재적용 2) 에이전트 라이선스 재검증 → safe mode 해제 3) 정책 재동기화(policy re-sync) 실행 4) 검사·차단 enforcement 복구 확인

7-2. 즉시 조치 — 복구 확인

갱신 후 license.log에서 entering safe mode·license invalid가 더 이상 발생하지 않고 policy re-sync ok가 기록되는지 확인한다. 에이전트 측에서 검사·차단이 실제로 동작하는지 표본 검증한다.

7-3. 배포 체크리스트

  • 근본 원인 규명 (라이선스 만료 → 안전모드 → 검사·차단·정책 비활성화)
  • 신규 라이선스 키 갱신·재적용 후 안전모드 해제 확인
  • 정책 재동기화 실행 및 policy re-sync ok 확인
  • 검사·차단 enforcement 표본 검증(테스트 파일로 탐지·차단 재현)
  • 만료 30일 전 자동 알림 · 안전모드 진입 경보 등록

8.재발 방지 & 모니터링 권고

  • 만료 사전 알림: 라이선스 만료 30일 전 자동 알림을 발송해 갱신 누락으로 인한 전면 비활성화를 원천 차단한다(SUP-146 권고와 동일).
  • 안전모드 경보: entering safe mode / license invalid / scan skipped 패턴을 알림 규칙으로 등록해 통제 공백을 즉시 감지한다.
  • 라이선스 대시보드: 잔여 유효기간을 지표화해 만료 임박을 상시 가시화한다.
  • PII 로깅 정리: license.log·admin-approve.log·dlp-approve-api.log의 이메일·연락처·IP 평문 적재를 마스킹으로 전환한다.
  • 이슈 연계: 본 건을 SUP-146과 linked issue로 기록해 동일 증상 재문의 시 참조한다.

10.부록 A — 에이전트 분석 파이프라인

본 보고서는 로컬 데몬(도구 실행·PII 마스킹·SLM 요약)과 원격 LLM(Claude Code CLI, claude-opus-4-8)이 결합된 자율 tool-use 루프의 산출물이다. 누적 도구 호출 5회, 종료 사유 end_turn.

11.부록 B — PII 마스킹 처리 내역

로컬 에이전트가 원격 LLM 전달 전 세 로그의 개인정보를 토큰으로 치환했다. pii.reset()은 케이스당 1회만 수행되며, 3개 로그를 한 세션에서 순서대로 마스킹해 토큰이 케이스 전역에서 유일하다(같은 원문 = 같은 토큰, 다른 원문 = 다른 토큰). 원문 PII는 원격 LLM에 전달되지 않는다.
